# Gitの基本の使い方

Git の環境が整ったら、いよいよ実際にコードを管理する流れを体験しましょう。
Git の操作は、最初は「なぜこんなに手順が多いの？」と不思議に思うかもしれません。でも、それぞれのステップには **「慎重にコードを育てる」** ための深い理由があります。

基本の 3 つのステップ：**「add」「commit」「push」** を中心に、Git の日常的なワークフローを解説します。

## 1. 変更を記録するまでの 3 つのエリア

Git には、あなたの変更が旅をする 3 つの場所があります。この構造を理解することが、Git 習得の最短ルートです。

1. **作業ディレクトリ (Working Directory)**: あなたが今、VS Code で編集している「ナマ」のファイル群です。
2. **ステージングエリア (Staging Area / Index)**: 次のコミット（記録）に含めるファイルを「準備」しておく場所です。
3. **リポジトリ (Repository / Local)**: 変更が正式に「記録」された場所です。

### ステップ A: `git add`（準備する）
```bash
git add main.ts
```
「このファイルの変更は、次の記録に含めたいな」と思ったら、ステージングエリアに移動させます。
全部のファイルを一度に `add` するのではなく、関係のある変更だけを細かく選んで `add` するのがプロの技です。

### ステップ B: `git commit`（記録する）
```bash
git commit -m "ログイン機能のバグを修正"
```
ステージングエリアにある変更を、リポジトリに永久保存します。このとき付ける **「コミットメッセージ」** は、未来の自分や仲間への手紙です。

### ステップ C: `git push`（共有する）
```bash
git push origin main
```
ローカルに溜まった記録を、GitHub などのリモートサーバーへ送り出します。これでようやく、他の人もあなたの変更を見ることができるようになります。

## 2. 状態を確認する魔法のコマンド：`git status`

Git を操作していて、「今、どうなってるんだっけ？」と不安になったら、すぐにこのコマンドを叩きましょう。

```bash
git status
```
- 赤い文字：変更したけど、まだ `add` していないファイル。
- 緑の文字：`add` したけど、まだ `commit` していないファイル。

Git は常にあなたの作業状況を監視し、次に何をすべきか（あるいは何を忘れているか）を教えてくれます。

## 低レイヤーの視点：HEAD と参照

Git の中では、今あなたが「どの地点」にいるのかを **HEAD** というポインタ（矢印）が指し示しています。

- **HEAD**: 「現在作業中のコミット」へのポインタ。
- **main (ブランチ)**: 「そのブランチの最新のコミット」を指す、動くラベル。

あなたがコミットをするたびに、`main` というラベルは最新のコミットへと自動的に動いていきます。Git における「ブランチ」とは、巨大なデータのコピーではなく、**「特定のコミットを指しているたった 40 文字のハッシュ値が書かれたテキストファイル」** に過ぎません。
この「参照（ポインタ）」を付け替えるだけでパラレルワールド（ブランチ）を切り替えられる。この驚異的な効率性が、Git の真髄です。

## 私の視点：コミットは「小さく、美しく」

初心者がやりがちなのが、丸一日作業して、何十箇所の変更をまとめて「いろいろ直した」というメッセージで一気にコミットすることです。
これは、後でバグが見つかったときに地獄を見ます。

理想は **「アトミックコミット」** です。
「一つの意味のある変更（例：ボタンの色を変えた、誤字を直した）」ごとにコミットする。そうすれば、もし失敗しても、その最小単位だけを取り消すことができます。
「きれいな履歴は、きれいな設計の裏返し」。コミットグラフが美しく一本の線、あるいは整然とした枝分かれになっているのを見ると、エンジニアとしての心が洗われますよ。

### まとめ

- 変更は「作業場所 ＞ 準備場所 (add) ＞ 記録場所 (commit)」と移動する。
- `git status` は、今どこにいるかを教えてくれる最強の味方。
- `git push` で GitHub に同期する。
- 裏側では `HEAD` やブランチという「ポインタ」が動いているだけ。
- コミットは細かく、メッセージは具体的に書くのがプロ。

### 展望

基本の使い方がわかったら、次はターミナル上で動く様々なツールに共通する「お作法」について学びましょう。Git 以外でも役立つ、エンジニアの「共通言語」としての CLI の知識です。
