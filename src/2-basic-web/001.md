# Webサイトの仕組み

私たちが毎日、iPhoneやPCのブラウザで何気なく見ているWebサイト。URLを入力してエンターキーを押すと、一瞬で画面に情報が表示されます。しかし、その「一瞬」の裏側では、世界中のコンピュータが驚くほど緻密な連係プレーを行っています。

プログラミングを学ぶ上で、この「情報の流れ」と「役割分担」を理解することは、単にコードを書くこと以上に重要な基礎となります。

## 登場人物：クライアントとサーバー

Webの世界は、大きく分けて2つの役割（役職）で成り立っています。これを **クライアント・サーバーモデル** と呼びます。

1. **クライアント (Client)**: サービスを「利用する」側。
   具体的には、あなたのブラウザ（Google Chrome, Safari, Microsoft Edgeなど）のことです。「クライアント」という言葉には「依頼人」という意味があり、まさにサーバーに対して「情報をください」と依頼する立場です。

2. **サーバー (Server)**: サービスを「提供する」側。
   Webサイトの本体（HTMLファイル、CSS、画像、動画など）を大切に保管し、依頼に応じてそれらを送り出す、24時間365日休まず動いているコンピュータです。

あなたがブラウザのURL欄に `https://google.com` と入力して実行した瞬間、あなたのコンピュータ（クライアント）から、Googleの巨大なデータセンターにあるコンピュータ（サーバー）へ、**「このページを見せてください！」** というメッセージが送られます。これを **リクエスト (Request)** と呼びます。

それに対し、サーバーが中身を確認して「はい、どうぞ！」とデータを送り返してくることを **レスポンス (Response)** と呼びます。

## HTTP：Webの共通言語

世界中の多種多様なコンピュータが会話するためには、共通の「言語」や「ルール」が必要です。それが **HTTP (HyperText Transfer Protocol)** です。

HTTPには、いくつかの決まった「型」があります。
- **メソッド**: 「取得したいのか（GET）」、「送信したいのか（POST）」といった意図を伝えます。
- **ステータスコード**: 「成功したよ（200 OK）」、「見つからなかった（404 Not Found）」、「エラーが起きた（500 Internal Server Error）」といった結果を3桁の数字で伝えます。

私たちがブラウザでページを読み込んでいるときは、主にこの「GETメソッド」を使ってサーバーに依頼を出しているのです。

## 低レイヤーの視点：IPアドレスとDNS

サーバーにリクエストを送るためには、そのサーバーがインターネット上の「どこ」にあるかを知る必要があります。コンピュータの世界では、これを **IPアドレス**（例: `142.250.196.110`）という数字の羅列で管理しています。

しかし、人間がこの数字を覚えるのは大変です。そこで、`google.com` のような「ドメイン名」を IPアドレスに変換してくれる仕組み、**DNS (Domain Name System)** が活躍します。

1. ブラウザが DNSサーバーに「google.com の IPアドレスを教えて！」と聞く。
2. DNSが「それは xxx.xxx.xxx.xxx です」と答える。
3. ブラウザがその IPアドレスに向けて HTTPリクエストを送る。

Webサイトが表示されるまでのわずかな時間には、これだけのステップが含まれているのです。

## ブラウザの仕事：レンダリングエンジン

サーバーから送られてくるレスポンス（データ）の中身は、実はただのテキスト（文字）の集まりです。そのまま見ても、私たち人間には意味が分かりません。

ブラウザの最も重要な仕事は、このテキストデータを解析し、色鮮やかな画面に描き出すことです。これを **レンダリング (Rendering)** と呼びます。

レンダリングは、大きく分けて以下の3ステップで行われます。

1. **解析 (Parsing)**: 送られてきた HTML や CSS を読み込み、コンピュータが扱いやすい内部的なデータ構造（DOM/CSSOM）に変換します。
2. **レイアウト計算 (Layout)**: どの要素がどのくらいの大きさで、画面のどこに配置されるべきかを計算します。
3. **描画 (Painting)**: 計算結果に基づいて、画面の1ピクセルずつに色を置いていきます。

現代のブラウザは、この一連の処理をミリ秒単位（1秒の1000分の1）で行っています。裏側では、**V8**（JavaScriptエンジン）や、**Blink** / **WebKit** といった巨大なレンダリングエンジンが、CPU や GPU をフル活用して動いています。

## 私の視点：Webは「分散」している

Webの面白さは、一つの画面を作るために、実は何十、何百というサーバーが協力しているところにあります。
例えば、ニュースサイトを開いたとき、文章はニュースサーバーから、画像は画像専用のサーバー（CDN）から、広告は広告配信サーバーから……といった具合に、裏側では並列でリクエストが飛び交っています。

この「分散したシステムを一つの体験としてまとめ上げる」のが、フロントエンドエンジニアである私たちの腕の見せ所なのです。

### まとめ

- Webは「クライアント（ブラウザ）」と「サーバー（コンピュータ）」のやり取り。
- 「リクエスト」を出して「レスポンス」を受け取るのが基本。
- やり取りのルールは「HTTP」。
- ブラウザは「レンダリング」によってテキストをビジュアルに変換する。
