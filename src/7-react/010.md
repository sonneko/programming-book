# AppRouterについて

Next.js 13 で登場し、現在主流となっている **App Router**。これは単なるルーティングの変更ではなく、React 誕生以来の最も巨大なアップデートである **「React Server Components (RSC)」** を実戦に投入するための、革命的なアーキテクチャです。

これまでの Web 開発は「いかにクライアント（ブラウザ）に仕事をさせるか」を追求してきましたが、App Router はそのベクトルを再び「サーバー」へと戻しました。

## 1. サーバーコンポーネント (RSC) という新境地

App Router では、すべてのコンポーネントはデフォルトで **「サーバーコンポーネント」** です。

```tsx
// app/page.tsx
// この関数は「サーバー」で実行される！
async function Page() {
  const data = await db.query('SELECT * FROM posts'); // 直接DBを叩ける！
  
  return (
    <div>
      {data.map(post => <Post key={post.id} {...post} />)}
    </div>
  );
}
```

サーバーコンポーネントの凄さは、**「ブラウザに JavaScript を送らなくていい」** という点にあります。
サーバーで HTML を作り、その計算に使ったライブラリやロジックはサーバーに残したまま、結果の HTML だけをブラウザに届けます。
これにより、ユーザーの PC で実行される JavaScript の量が劇的に減り、サイトの読み込みが魔法のように速くなります。

## 2. サーバーとクライアントの「共存」

もちろん、ボタンを押したときの反応やアニメーションなど、ブラウザで動かしたい部品もあります。その時は、ファイルの先頭に **`'use client'`** と書くことで、その部品だけを **「クライアントコンポーネント」** に切り替えることができます。

「重いデータ処理はサーバーで」「軽快な操作はブラウザで」。この使い分けを一つのプロジェクト、一つのファイル構造の中でシームレスに行えるのが App Router の強みです。

## 低レイヤーの視点：ストリーミングと Suspense

App Router は、**「ストリーミング」** という技術を使って、画面をバラバラに届けることができます。

例えば、サイドバーはすぐに出せるけど、メインの記事一覧は DB からの取得に時間がかかる……という場合。
これまでは「全部揃うまで真っ白」でしたが、App Router なら「サイドバーだけ先に送って、記事部分は『読込中...』と出しておき、準備ができた順に流し込む」ということが可能です。
これは、HTTP 通信のパイプを繋ぎっぱなしにして、データが届くたびに HTML のパーツを少しずつブラウザにプッシュする、という非常に高度な通信制御を行っています。

## 私の視点：アーキテクチャの回帰と進化

サーバーで HTML を作って送る、という手法は PHP や Ruby on Rails が 20 年前からやっていたことです。
でも、App Router はただの先祖返りではありません。
「サーバーサイドの堅牢さ」と「モダンフロントエンドの操作感」を、React という共通の言語で融合させた、**Web 開発の「完成形」の一つ** だと私は感じています。

最初はサーバーとクライアントの境界線に戸惑うかもしれませんが、この「ハイブリッドな感覚」に慣れてくると、もう以前のやり方には戻れなくなりますよ。

### まとめ

- App Router は React Server Components を前提とした新世代のルーティング。
- サーバーコンポーネントは JavaScript をクライアントに送らず、読み込み速度を爆速にする。
- 直接 DB やセキュリティの高い API を叩けるため、バックエンドとの境界が溶けていく。
- `'use client'` を使って、必要な部分だけをインタラクティブにできる。
- ストリーミングによって、準備ができたパーツから順次表示できる。

### 展望

App Router という強力な基盤を手に入れたら、次はそこで使われる **「特別な関数」** たち――データの取得やキャッシュ、メタデータの管理を司る関数群をマスターしましょう。
