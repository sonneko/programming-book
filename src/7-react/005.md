# フック（Hooks）のアーキテクチャ

React のコンポーネントは「ただの関数」です。関数は、実行が終わればその中の変数は消えてしまいます。それなのに、なぜ React のコンポーネントは「前のカウント値」を覚えていたり、「画面が表示された後に通信する」といった複雑なことができるのでしょうか？

その秘密が **Hooks (フック)** です。
Hooks は、関数コンポーネントという「ただの関数」の中に、React の内部システムへアクセスするための「釣り針（フック）」を引っ掛けるような仕組みです。

## 1. useState：記憶を持つためのフック

最もよく使うのが `useState` です。

```tsx
const [count, setCount] = useState(0);
```

この 1 行で、React はあなたのコンポーネント専用の「メモリ空間」を確保します。
`setCount` を呼ぶと、React は「あ、このコンポーネントの状態が変わったな」と検知し、もう一度関数を実行（再レンダリング）します。このとき、`useState(0)` は 0 ではなく、**「保存されていた最新の値」** を返してくれるのです。

## 2. useEffect：副作用（きっかけ）のためのフック

「画面が表示されたときに一度だけ実行したい」「データが変わったときにログを出したい」といった、レンダリング以外の処理（副作用）を扱うのが `useEffect` です。

```tsx
useEffect(() => {
  console.log("画面に現れました！");

  // クリーンアップ関数（消えるときに実行）
  return () => console.log("消えます...");
}, []); // この [] が「いつ実行するか」の鍵
```

## 低レイヤーの視点：実行順序という「制約」

Hooks には **「ループや条件分岐の中で呼んではいけない」** という絶対のルールがあります。なぜでしょうか？

実は、React の内部では、各コンポーネントの Hooks が **「配列（あるいは連結リスト）」** として管理されています。

1. 1 回目のレンダリング：`useState` が呼ばれた。配列の 0 番目に値を保存。
2. 2 回目のレンダリング：また `useState` が呼ばれた。配列の 0 番目から値を取り出す。

もし `if` 文の中で Hooks を呼んでしまうと、前回は 3 つ呼ばれたのに今回は 2 つしか呼ばれない……といったことが起きます。すると React は「あれ？ 2 番目にあったはずの状態が 1 番目に来ているぞ！」とパニックになり、状態がぐちゃぐちゃに混ざってしまいます。
**「常に同じ順番で呼ばれること」**。これが、Hooks が魔法を実現するための、物理的な代償なのです。

## 私の視点：Hooks は「ロジックの革命」である

Hooks が登場する前の React（クラスコンポーネント）では、ロジックを再利用するのは至難の業でした。
Hooks の本当の凄さは、**「UI とは関係ない純粋なロジックを、関数として切り出せるようになったこと」** にあります。

「データの取得」「フォームの入力管理」「タイマーの制御」。
これらを Hooks として切り出し、複数のコンポーネントで共有する。この **「カスタムフック」** という武器を手に入れたことで、React は圧倒的な開発効率を手に入れたのです。

### まとめ

- Hooks は関数コンポーネントに「状態」や「ライフサイクル」を与える仕組み。
- `useState`: 状態を保存し、変更時に再レンダリングを起こす。
- `useEffect`: レンダリングとは別の「副作用」を適切なタイミングで実行する。
- **実行順序が命**: 内部でリスト管理されているため、条件分岐やループ内での使用は厳禁。
- カスタムフックを使うことで、ロジックを部品化して再利用できる。

### 展望

Hooks の仕組みがわかったら、次はそれらを組み合わせて自分だけの魔法を作る **「カスタムフック」** の具体的な設計手法について学んでいきましょう。
