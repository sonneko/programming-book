# SPA（Single Page Application）の構造と価値

かつての Web サイトは、リンクをクリックするたびに画面が白くなり、数秒待ってから新しいページが表示されるのが当たり前でした。しかし、現代の Twitter(X) や Gmail を見てください。ページ全体が切り替わるのではなく、アプリのようにサクサクと必要な部分だけが変化します。

この体験を実現しているのが **SPA (Single Page Application)** です。
React が真価を発揮するのは、まさにこの SPA という形態のアプリケーションを作る時です。

## 1. SPA の仕組み：リロードしないという革命

従来の Web サイト（MPA: Multi Page Application）と SPA の最大の違いは、**「誰が画面を作っているか」** です。

- **従来のサイト (MPA)**: リンクを踏むたびにサーバーに「新しい HTML をください」と頼みます。サーバーは 0 から HTML を作り、ブラウザはそれを受け取って画面全体を描き直します。
- **SPA**: 最初に 1 回だけ、空っぽに近い HTML と大量の **JavaScript** を読み込みます。その後の画面遷移は、JavaScript が DOM を直接書き換えることで行います。サーバーからは「データ（JSON）」だけを最小限に受け取ります。

「ページは 1 枚（Single Page）だけで、中身を入れ替え続けている」から、SPA なのです。

## 2. メリット：滑らかさと通信量の削減

SPA にすることで、劇的なメリットが生まれます。

1. **UX (ユーザー体験) の向上**: ページ全体のリロード（白い画面）がないため、ネイティブアプリのようなヌルヌルとした操作感が得られます。
2. **サーバー負荷の低減**: 重い HTML を毎回送る必要がなくなり、純粋なデータだけをやり取りすれば良くなります。
3. **状態の維持**: 音楽を再生しながら別のページに移動する、といった「ページを跨いだ状態の保持」が可能になります。

## 低レイヤーの視点：History API とルーティングの擬態

「ページ遷移をしていないのに、なぜ URL が変わるの？」と不思議に思いませんか？
実は、SPA はブラウザの **History API** という機能を使って、**「ページ遷移をしたふり」** をしています。

通常、URL を書き換えるとブラウザはリロードしてしまいますが、`history.pushState` というコマンドを使うと、リロードさせずに URL だけを書き換え、履歴に残すことができます。
ブラウザの「戻る」ボタンが押されたときも、SPA はそれを検知して JavaScript で画面を元の状態に戻します。
つまり、SPA はブラウザというプラットフォームの上に構築された、壮大な「手品」のようなシステムなのです。

## 私の視点：SPA の代償（SEO と初期読み込み）

SPA は最高に見えますが、タダではありません。

一番の弱点は、**「最初の読み込みが重い」** ことです。
アプリの全機能が入った巨大な JavaScript ファイルを最初にダウンロードする必要があるため、ネット環境が悪いと「最初の画面が出るまで」に時間がかかります。
また、初期状態が空っぽの HTML なので、検索エンジンのロボット（クローラー）が内容を正しく読み取れない、という SEO 上の問題もかつてはありました。

これらの弱点を克服するために、**Next.js** のような「サーバーでもレンダリングする (SSR)」技術が生まれました。技術は常に、課題と解決の繰り返しで進化しているのです。

### まとめ

- SPA はページ遷移をリロードなしで行い、アプリのような体験を提供する。
- JavaScript が DOM を直接書き換えることで、高速な画面更新を実現する。
- サーバーとはデータ (JSON) だけをやり取りする。
- History API を使うことで、リロードなしで URL を変更している。
- 初期読み込みの重さや SEO が課題だが、SSR などの技術で解決されつつある。

### 展望

SPA の概念がわかったら、次は具体的に「どうやって URL と画面を連動させるか」を解決するライブラリ **React Router** について学びましょう。
