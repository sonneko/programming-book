# データ志向

プログラミングの学び始めに、多くの人が「オブジェクト指向（OOP）」に出会います。そこでは「データと振る舞いを一つの『オブジェクト』にまとめよう」と教わります。しかし、関数型プログラミングの世界観は、それとは正反対の場所にあります。

それが **「データ志向（Data-Oriented）」** という考え方です。

## 「モノ」ではなく「流れ」で考える

オブジェクト指向がプログラムを「意思を持ったモノ（オブジェクト）の集まり」として捉えるのに対し、関数型プログラミング（およびデータ志向）は、プログラムを **「データが加工されていくパイプライン」** として捉えます。

私がよく使う比喩は「粘土細工」と「工場のライン」です。

- **オブジェクト指向**: 粘土（データ）をこねる人（メソッド）が、粘土自身の中に住んでいるイメージ。
- **データ志向**: 原材料（データ）がベルトコンベアに乗って流れていき、各工程（関数）で形を変えられ、最後に製品（結果）になるイメージ。

この「データ」と「処理（関数）」を完全に切り離すことが、データ志向の核心です。

## なぜ分離するのか？

データと処理を分けることで、いくつかの素晴らしいメリットが生まれます。

### 1. データの構造が透明になる

オブジェクト指向では、データ（フィールド）がカプセル化（隠蔽）されていることが美徳とされます。しかし、関数型ではデータは「ただのデータ」として、ありのままの姿で公開されます。これにより、どのようなデータが流れているのかを把握しやすくなります。

### 2. 処理の組み合わせが自由自在

データが「ただのリスト」や「ただのレコード」であれば、それを受け取って処理する関数を無限に作ることができます。特定のクラスに属するメソッドである必要がないため、既存の関数を組み合わせて新しい処理を作るのが非常に簡単です。

```typescript
// TypeScriptでのデータ志向的なアプローチ
interface User {
  id: number;
  name: string;
  age: number;
}

const users: User[] = [/* ... */];

// データを変換する純粋な関数たち
const isAdult = (user: User) => user.age >= 20;
const getName = (user: User) => user.name;

// パイプラインとして記述できる
const adultNames = users
  .filter(isAdult)
  .map(getName);
```

このコードでは、`User` というデータ構造と、`isAdult` や `getName` という処理が独立しています。後から `sortByAge` という関数を追加するのも、`User` クラスを書き換える必要はなく、ただ新しい関数を書くだけで済みます。

## 低レイヤーの視点：メモリレイアウトとキャッシュ効率

実は「データ志向」という言葉は、関数型プログラミングの文脈だけでなく、ゲームエンジン開発などの超高速な処理が求められる分野でも使われます（*Data-Oriented Design*）。

そこでの関心事は、CPUの「キャッシュメモリ」です。

オブジェクト指向のように、メモリ上のあちこちに散らばった「オブジェクト」を追いかけるよりも、同じ種類のデータをメモリ上にギュッと並べて（配列など）、それを一気に処理する方が、現代のCPUにとっては圧倒的に高速なのです。

「データと処理を分ける」という思想は、人間にとっての理解のしやすさだけでなく、ハードウェアにとっても効率的な構造を生み出す可能性を秘めています。

## 私の視点：不変性（Immutability）との相性

データ志向を実践する上で欠かせないのが「不変性」です。流れているデータを直接書き換えてしまうと、パイプラインの途中で何が起きたか分からなくなります。

データを「変更」するのではなく、常に「新しい状態のデータを作成して次に渡す」。このルールを守ることで、プログラムの予測可能性は飛躍的に高まります。私が書いたプログラムがバグったとき、データの「流れ」を追いかけるだけで原因が特定できるのは、本当に心地よい体験です。

### まとめ

- データ志向は「データ」と「処理（関数）」を明確に切り離す。
- プログラムを、データの変換処理が連なる「パイプライン」として捉える。
- このアプローチは、可読性の向上だけでなく、実行効率の向上にも寄与する。
