# カリー化

「カリー化（Currying）」――。初めてこの名前を聞いたとき、私は「えっ、カレー（Curry）？」と美味しそうな想像をしてしまいました。でも、実はこの名前、数理論理学者のハスケル・カリー（Haskell Curry）氏に由来しています。

カリー化は、関数型プログラミングにおいて「関数のパーツ化」を極限まで押し進めるための、非常にエレガントなテクニックです。

## カリー化とは何か？

一言で言えば、カリー化とは**「複数の引数をとる関数を、『引数を1つだけとる関数』の連鎖に作り変えること」**です。

例えば、2つの数値を足す `add(x, y)` という関数を考えてみましょう。

```python
# 普通の関数（Python）
def add(x, y):
    return x + y

print(add(2, 3)) # 5
```

これをカリー化すると、以下のようになります。

```python
# カリー化された関数
def add_curried(x):
    def inner(y):
        return x + y
    return inner

# 呼び出し方
add_two = add_curried(2) # まず「2を足す関数」ができる
result = add_two(3)      # それに3を渡すと5になる
print(result) # 5

# まとめて呼ぶ場合
print(add_curried(2)(3)) # 5
```

「わざわざ面倒な書き方をしなくても……」と思うかもしれませんが、この `add_two` のように「引数の一部だけを先に渡して、新しい関数を作る」ことができるのが、カリー化の最大のメリットです。これを**部分適用（Partial Application）**と呼びます。

## なぜカリー化が嬉しいのか？

カリー化の本質は「ロジックの再利用」にあります。

例えば、Web開発で「特定のログレベルを付けてメッセージを出力する関数」を作りたいとします。

```javascript
// JavaScriptでの例
const logger = (level) => (message) => {
  console.log(`[${level}] ${message}`);
};

const infoLog = logger("INFO");
const errorLog = logger("ERROR");

infoLog("システムを起動しました"); // [INFO] システムを起動しました
errorLog("メモリが不足しています"); // [ERROR] メモリが不足しています
```

このように、汎用的な関数（`logger`）から、特定の用途に特化した関数（`infoLog`）を量産できるのです。これは、関数を「部品」として組み合わせていく関数型プログラミングの思想と完璧にマッチします。

`add_curried(2)` を実行したとき、内部で作られた関数（`inner`）は、その時の変数 `x=2` を「記憶」したまま関数の外に飛び出します。コンピュータのメモリ上では、この `inner` 関数のコードへのポインタと、記憶された変数 `x` のセットが保持されています。

コンパイラやインタプリタは、この「関数の種」のようなデータをヒープ領域に確保し、後で呼び出されたときに `x` を取り出せるように管理しているのです。

## Haskellにおけるカリー化

Haskellという言語では、なんと**デフォルトですべての関数がカリー化されています**。

```haskell
-- Haskellでは
add x y = x + y

-- これを add 2 3 と呼ぶのは、内部的には (add 2) 3 と同じ
```

Haskellの関数の型定義を見ると `Int -> Int -> Int` のようになっています。これは「Intを受け取って、『Intを受け取ってIntを返す関数』を返す」という意味です。最初からカリー化が前提の設計になっているため、関数の組み合わせが非常にスムーズに行えます。

## 私の視点：数学的な美しさ

カリー化を理解すると、世界が少し違って見えます。「3つの値を処理する」という複雑な現象を、「1つの値を処理する」という単純なステップの積み重ねに分解できるからです。これは、複雑な問題を小さな問題に分けるというエンジニアリングの基本原則そのものです。

最初は `f(x)(y)(z)` という書き方に違和感を覚えるかもしれませんが、慣れてくると「関数を特化させていく」プログラミングの楽しさに気づけるはずです。

### まとめ

- カリー化は、多引数関数を「1引数関数の連鎖」に変えること。
- 「部分適用」によって、汎用的な関数から専門的な関数を簡単に作れる。
- その裏側では「クロージャ」というメモリ管理の仕組みが動いている。

### 展望

次は、関数型プログラミングのもう一つの柱である「データ志向」について考えてみましょう。プログラムを「モノ」の集まりではなく、「データの流れ（パイプライン）」として捉える視点を手に入れます。
