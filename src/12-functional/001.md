# 純粋とは

「純粋（Pure）」という言葉を聞くと、皆さんは何を思い浮かべますか？ 汚れのない、混じりけのない……そんなイメージかもしれません。関数型プログラミングの世界でも、この「純粋さ」は非常に重要なキーワードです。

私が初めてこの概念に出会ったとき、正直に言って「そんなの当たり前じゃない？」と思いました。でも、深く学んでいくうちに、私たちが普段書いているプログラムがいかに「不純」であるか、そして「純粋」であることがどれほど強力な武器になるかを思い知らされたのです。

## 数学的な「関数」の本来の姿

プログラミングにおける「関数」は、数学における「関数」とは少し違います。数学の $f(x) = x + 1$ という関数を考えてみてください。$x$ に $5$ を入れれば、世界中のどこで、いつ計算しても、答えは必ず $6$ になります。$x$ が $5$ なのに、昨日は $6$ だったけど今日は気分で $7$ になる、なんてことはあり得ません。

しかし、プログラミングの世界ではどうでしょうか。

```python
# 不純な関数の例
count = 0
def increment_counter(x):
    global count
    count += 1
    return x + count

print(increment_counter(5)) # 6
print(increment_counter(5)) # 7
```

この `increment_counter` は、同じ `5` という引数を渡しているのに、呼び出すたびに結果が変わります。これは関数の外部にある `count` という状態に依存し、かつそれを書き換えているからです。これを「副作用（Side Effect）」と呼びます。

「純粋」とは、このような副作用が一切ない状態を指します。

## 純粋関数の二つの条件

ある関数が「純粋」であるためには、以下の二つの条件を満たす必要があります。

1. **参照透過性（Referential Transparency）**: 同じ入力に対して、常に同じ出力を返すこと。
2. **副作用がないこと**: 関数の外部の状態を変化させないこと。

具体的には、以下のような操作を行わないことが求められます。
- グローバル変数や外部変数の書き換え。
- 引数として渡されたオブジェクトや配列の直接的な変更（破壊的な操作）。
- 標準出力へのプリントやファイルの読み書き（I/O）。
- 現在時刻の取得や乱数の生成（これらは呼び出すたびに結果が変わるため不純です）。

## なぜ「純粋」を追求するのか？

「えっ、プリントもできないし乱数も使えないなんて、何もできないじゃん！」と思うかもしれません。確かに、プログラムのどこかでは副作用（画面に表示する、DBに保存するなど）が必要です。しかし、プログラムの「ロジック」の部分を純粋関数として切り出すことで、驚くべきメリットが得られます。

### 1. テストの容易性

不純な関数のテストは地獄です。外部のDBの状態を整えたり、ネットワークをシミュレート（モック）したりする必要があります。しかし、純粋関数なら「引数」と「戻り値」だけを見れば良いのです。状態の準備が必要ないので、テストコードは非常にシンプルになります。

### 2. 計算のキャッシュ（メモ化）

同じ入力なら必ず同じ出力が返ることが保証されているため、計算結果を保存（キャッシュ）しておくことができます。これを*Memoization*（メモ化）と呼びます。重い計算を伴う関数でも、一度計算してしまえば二回目以降は一瞬で終わります。

### 3. 並列処理の安全性

マルチコアCPUをフル活用しようとすると、複数のスレッドが同時に同じデータを触ることで発生する「競合」が問題になります。しかし、純粋関数は外部の状態を一切書き換えないため、複数のスレッドで同時に実行しても全く問題が起きません。

## 低レイヤーの視点：コンパイラ最適化

「純粋さ」は人間だけでなく、コンピュータ（コンパイラ）にとっても嬉しい性質です。コンパイラが「この関数は純粋だ」と判断できれば、以下のような最適化が可能になります。

- **定数畳み込み**: プログラムを実行する前に、計算結果をあらかじめ計算して埋め込んでおく。
- **デッドコード削除**: 関数の戻り値がどこでも使われていない場合、副作用がないことが確実なら、その関数呼び出し自体を消去して実行時間を短縮できる。

## 私の視点：純粋さは「予測可能性」

結局のところ、私が純粋関数を愛しているのは「裏切らないから」です。関数のコードを読んでいるとき、その中身だけを見れば挙動が100%理解できる。外部のどこかで何かが変わっているかも……と疑心暗鬼になる必要がない。この「認知負荷の低さ」こそが、大規模なソフトウェアを開発する上での最大の救いになります。

皆さんも、まずは小さな関数から「純粋」に書くことを意識してみてください。きっと、コードが以前よりずっと透き通って見えるはずです。

### まとめ

- 純粋関数は「同じ入力には同じ出力」と「副作用なし」を徹底する。
- 副作用をロジックから分離することで、テストや並列化が容易になる。
- 最終的なアウトプット以外の部分は、できるだけ純粋に保つのが良い設計。

### 展望

次は、この純粋関数を極限まで突き詰めた言語「Haskell」の世界を覗いてみましょう。副作用をどのように「管理」しているのか、その不思議な仕組みに触れていきます。
