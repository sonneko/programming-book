# 3Dを支える数学

「3Dプログラミングには数学が必要だ」と聞いて、身構えてしまう人もいるかもしれません。確かに、本格的な物理シミュレーションやレンダリングエンジンの開発には高度な数学が使われます。しかし、3Dの空間で物体を動かし、画面に映し出すために必要な「基本の道具」は、実はそれほど多くありません。

今回は、エンジニアとして3Dを扱う上で絶対に欠かせない **ベクトル** と **行列**、そして **座標変換** について解説します。

### ベクトル：向きと大きさ

3D空間における「点」や「方向」を表現するのが **ベクトル** (*Vector*) です。
3D空間なので、通常は `(x, y, z)` という3つの成分を持ちます。

- **位置としてのベクトル**: 原点（0, 0, 0）から見て、どこに物があるか。
- **方向としてのベクトル**: どちらを向いているか、どれくらいの速さで動いているか。

3Dプログラミングでよく使う演算には、以下のようなものがあります。
- **加算・減算**: 物体を移動させたり、2点間の距離を測ったりします。
- **正規化** (*Normalization*): ベクトルの「大きさ」を1にすること。純粋に「向き」だけを知りたいときに使います。
- **内積** (*Dot Product*): 2つのベクトルの「重なり具合」を計算します。これを使うと、2つのベクトルのなす角度や、光がどれくらい正面から当たっているかを判定できます。
- **外積** (*Cross Product*): 2つのベクトルに対して「垂直なベクトル」を計算します。面の向き（法線）を求めるのに必須です。

### 行列：空間を歪ませる魔法の表

ベクトルが「点」なら、**行列** (*Matrix*) は「空間そのものを変形させるルール」です。
3Dでは主に 4x4 の正方行列が使われます。なぜ3成分なのに 4x4 なのかという理由は、平行移動・回転・拡大縮小をすべて一つの「掛け算」で表現するためです（これを *同次座標系* と呼びます）。

行列を使えば、以下のような操作がただの掛け算（行列 × ベクトル）で実現できます。
- **平行移動** (*Translation*): 物体を動かす。
- **回転** (*Rotation*): 物体を回す。
- **拡大縮小** (*Scaling*): 物体の大きさを変える。

これらを組み合わせた「モデル行列」を作れば、どんな複雑な配置も一つの行列で表現できてしまいます。さらに、行列同士をあらかじめ掛けておくことで、「腕を回してから、体全体を動かす」といった階層的な動きも、最終的な一つの行列に凝縮してGPUに渡すことができます。

### 座標変換の旅

3Dモデルが画面（2D）に映し出されるまでには、いくつかの座標系を旅します。これを **座標変換** と呼びます。

1. **ローカル座標** (*Local Space*): 3Dモデル自身の中心を原点とした座標。
2. **ワールド座標** (*World Space*): ゲームの世界全体の原点から見た座標。モデル行列を掛けて変換します。
3. **ビュー座標** (*View Space*): 「カメラ」から見た座標。カメラを原点に持ってくるための変換です。
4. **クリップ座標 / スクリーン座標**: 3D空間を2Dの画面に投影 (*Projection*) した座標。遠くのものを小さく見せる「透視投影」などがここで行われます。

### 3Dの回転とクォータニオン

「回転」は3D数学の中で最も厄介な部分の一つです。単純な「X軸、Y軸、Z軸の回転角度（オイラー角）」で計算しようとすると、特定の条件で回転軸が重なって動けなくなる **ジンバルロック** という現象に悩まされます。

これを解決するために使われるのが **クォータニオン** (*Quaternion* / 四元数) です。
複素数を拡張したような数学的構造で、ジンバルロックを防ぎ、滑らかな回転の補間（アニメーション）を実現できます。中身を数学的に完全に理解するのは大変ですが、エンジニアとしては「回転を安全に、滑らかに扱うための道具」として使いこなせれば十分です。

### 数学は「低レイヤーの共通言語」

ライブラリ（Three.jsやUnityなど）を使っていれば、これらの数学を直接意識することは少ないかもしれません。しかし、シェーダーを書いたり、パフォーマンスを最適化したり、あるいは独自のゲームエンジンを作ろうとしたとき、数学は強力な武器になります。

「なぜこのキャラクターは変な方向に飛んでいくのか？」「なぜライティングが不自然なのか？」
その答えは、常にベクトルや行列の計算の中にあります。数学は、バグを見つけるためのデバッグ能力そのものなのです。
