# 3Dモデルとテキストの描画

GPUの並列計算と数学の基礎を学んだところで、次は「具体的にどうやって複雑な形や文字を画面に出すのか」について考えてみましょう。

### 3Dモデリング：ポリゴンという単位

3D空間にある物体は、ほとんどの場合、小さな「三角形」の集まりで表現されます。これを **ポリゴン** と呼びます。なぜ四角形ではなく三角形なのでしょうか？ それは、どんなに複雑な面であっても、3つの点があれば必ず一つの平面が定義でき、計算が最もシンプルで済むからです。

1. **メッシュ** (*Mesh*): 三角形の集まり。頂点（座標）、法線（向き）、UV座標（テクスチャのどこを貼るか）のデータを持っています。
2. **マテリアル** (*Material*): そのメッシュが「どんな質感か」を定義します。色、反射率、透明度、あるいはシェーダーそのものが含まれます。
3. **テクスチャ**: メッシュの表面に貼り付ける画像データです。

私たちがHydentで3Dを扱うなら、Blenderなどの3D制作ソフトで作られたデータを `glTF` (*GL Transmission Format*) などの標準形式で読み込むことになるでしょう。glTFは「3D界のJPEG」を目指して作られた、Webでも扱いやすい効率的な形式です。

### テキスト描画：実はグラフィックスの難所

意外かもしれませんが、3Dグラフィックスにおいて「文字をきれいに描く」のは非常に難しい課題の一つです。

通常のWeb開発なら `<div>こんにちは</div>` と書けばブラウザが勝手に描いてくれますが、GPUのキャンバス（WebGL/WebGPU）に直接文字を描くには、一工夫必要です。

主な手法には以下のようなものがあります。
- **ビットマップフォント**: 全ての文字を一つの大きな画像（フォントアトラス）にまとめておき、必要な部分を四角い板ポリゴンに貼り付けて表示します。高速ですが、拡大するとボケてしまいます。
- **SDF** (*Signed Distance Field*): 文字の輪郭からの距離を計算したデータを使う手法。拡大しても輪郭が非常にシャープで、最近のゲーム開発では主流です。
- **ベクター描画**: 文字の輪郭線をパスとして扱い、リアルタイムで塗りつぶし計算を行います。最も高品質ですが、GPU負荷が高くなりがちです。

### 3Dモデリングの「低レイヤー」な見方

3Dデータを扱うとき、エンジニアは単に「形」として見るのではなく、「メモリ上のバッファ」として捉える必要があります。

- **VBO** (*Vertex Buffer Object*): 頂点データを並べたメモリ空間。
- **IBO** (*Index Buffer Object*): どの頂点を使って三角形を作るかという、頂点の番号（インデックス）を並べたメモリ空間。

GPUに描画を依頼するときは、これらのバッファをGPUメモリへ転送し、「ここからここまでのデータを使って描画して！」という命令を出します。このとき、データの並び方（メモリレイアウト）が1バイトでもズレていると、画面には恐ろしいノイズや破綻した図形が表示されます。

この「データの並び」への厳密さは、C言語やRustなどの低レベルプログラミングに近い感覚です。

### モデリングとエンジニアリングの境界

現代の3D開発では、プログラムで動的に形を作る「プロシージャル・モデリング」という手法もよく使われます。例えば、迷宮の地形を自動生成したり、風に揺れる草を一万本生成したりする処理です。

これはアーティストが手で作るのではなく、エンジニアが数式（ノイズ関数など）を使ってGPU上で頂点を生成します。ここでは、これまでに学んだ数学とシェーダーの知識が、そのまま「形を作る力」に直結します。

### まとめ

- 3Dモデルは三角形（ポリゴン）の集合体。
- glTF形式などを使って、外部データを効率的に読み込む。
- テキスト描画はSDFなどの高度なテクニックが必要な難所。
- 3Dデータの本質はメモリ上の「バッファ」である。

次は、これらの物体を動かし、相互作用させるためのプログラム全体の構造、つまり「メインループ」や「ゲームエンジンの設計」について見ていきましょう。
