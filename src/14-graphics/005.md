# GLSL：GPUとの対話言語

GPUという数千人の作業員たちに命令を送るには、JavaScriptやPythonとは別の、彼ら専用の言葉が必要です。それが **GLSL** (*OpenGL Shading Language*) です。

GLSLで書かれたプログラムを **シェーダー** (*Shader*) と呼びます。文法自体はC言語に非常によく似ていますが、その実行モデルや考え方は、通常のプログラミング言語とは全く異なります。

### シェーダー：一つの「式」を数千回実行する

通常のプログラミングでは、`for` ループを使ってデータを順番に処理します。

```javascript
// JavaScriptのイメージ
for (let i = 0; i < pixels.length; i++) {
  pixels[i].color = calculateColor(pixels[i]);
}
```

しかし、GLSLには（基本的には）データの集合をループで回すという概念がありません。GLSLプログラムは、**「一つのデータ（頂点やピクセル）に対して何をするか」** という関数だけを書きます。GPUはその関数をコピーし、数千個のコアで、すべてのデータに対して一斉に（並列に）実行します。

### GLSLの基本データ型

GLSLは、グラフィックス計算に特化した強力な型を持っています。

- **`float`**: 浮動小数点数。グラフィックスでは整数よりも圧倒的に多用されます。
- **`vec2`, `vec3`, `vec4`**: ベクトル型。`vec3(x, y, z)` のように座標や色をまとめて扱います。
- **`mat2`, `mat3`, `mat4`**: 行列型。3Dの座標変換に必須です。
- **`sampler2D`**: 画像（テクスチャ）データ。

面白いのは、ベクトル演算が標準でサポートされていることです。例えば `color * 0.5` と書くだけで、RGBすべての成分を半分にしてくれます。

### 修飾子：データはどこから来るのか

シェーダーは単体では動けません。CPU（JavaScript）からデータを受け取る必要があります。そのための「入り口」を修飾子で定義します。

1. **`attribute` (または `in`)**:
   頂点ごとに異なるデータ（座標、色など）。頂点シェーダーで使います。
2. **`uniform`**:
   すべての実行で共通のデータ（今の時刻、カメラの行列、マウスの位置など）。
3. **`varying` (または `out`)**:
   頂点シェーダーからフラグメントシェーダーへ渡されるデータ。GPUが自動的に、頂点間の値を補完（グラデーション）してくれます。

### 思考のパラダイムシフト：座標は「自分の居場所」

シェーダープログラミングにおける最大の難関は、「今、自分がどのデータ（ピクセル）を処理しているか」という視点を持つことです。

フラグメントシェーダーの中では、あなたは画面全体のどこか一箇所にいる「小さな作業員」になります。隣のピクセルが何色かは分かりません。自分に与えられた座標（UV座標など）だけを頼りに、「この座標なら、私はこの色になるべきだ」と計算するのです。

この「独立性」こそが、GPUの圧倒的な並列処理を可能にしています。

### 低レイヤーの現実：精度とパフォーマンス

GLSLでは、数値の計算精度（`highp`, `mediump`, `lowp`）を自分で指定できることがあります。

- **`highp`**: 高精度。美しい描画。
- **`lowp`**: 低精度。高速でメモリ消費が少ない。

モバイル端末などで非常に複雑なシェーダーを動かす場合、すべてを最高精度にするとカクついてしまいます。シニアエンジニアは「ここは色の変化が少ないから低精度でいいな」といった、ハードウェアに寄り添った調整を行います。

### まとめ

- GLSLはGPUのための並列処理専用言語。
- 「ループ」ではなく「一斉変換」の思考が必要。
- ベクトルや行列を扱う強力な型が標準搭載されている。
- データの入り口（修飾子）を正しく定義してCPUと連携する。

GLSLの言葉が分かれば、いよいよその言葉を使って、形を歪ませ、世界に色を塗る具体的な方法を学べます。次は「頂点シェーダー」と「フラグメントシェーダー」それぞれの役割を深く掘り下げてみましょう。
