# 質感の表現：テクスチャとマテリアル

3Dモデルの「形」が頂点と三角形でできているなら、その「見た目」を支配するのが **テクスチャ** (*Texture*) と **マテリアル** (*Material*) です。ただの灰色のポリゴンの塊が、滑らかな肌、冷たい金属、あるいは燃え盛る炎に見えるのは、すべてこのレイヤーの魔法によるものです。

### テクスチャ：デジタルな皮膚

テクスチャは、最も単純には「3Dモデルの表面に貼り付ける画像」のことです。
しかし、現代のグラフィックスにおいて、テクスチャは単なる「絵」以上の役割を持っています。

- **アルベド/ディフューズマップ**: 物体そのものの色。
- **ノーマル（法線）マップ**: 表面の細かい凹凸の情報。これを使うと、ポリゴンを増やさずに表面をザラザラに見せることができます。
- **ラフネスマップ**: 表面の粗さ。ツルツルか、ザラザラか。
- **メタルネスマップ**: その場所が金属か、そうでないか。

これらを重ね合わせることで、物質の複雑な質感が表現されます。

### サンプリング：ピクセルから色を「汲み出す」

テクスチャをモデルに貼る際、頂点には **UV座標** という「画像のどこを使うか」という情報（0.0から1.0の範囲）が持たされています。

フラグメントシェーダーは、このUV座標を使ってテクスチャから色を取り出します。これを **サンプリング** と呼びます。
ここで一つ問題が起きます。テクスチャの解像度と、実際に画面に表示されるサイズが一致しない場合です。

- **拡大時**: ピクセルとピクセルの間をどう埋めるか（線形補間など）。
- **縮小時**: 遠くにある物体でテクスチャがチラつく「モアレ」をどう防ぐか。

これを解決するのが **ミップマップ** (*Mipmap*) です。あらかじめ縮小した画像を何段階か用意しておき、距離に応じて最適なものに切り替える。この小さな工夫が、滑らかで美しい映像を実現しています。

### マテリアル：光と物質の対話

マテリアルは、テクスチャや数値を組み合わせて「その物体が光に対してどう反応するか」を定義した仕組みです。

現代の主流は **PBR** (*Physically Based Rendering* / 物理ベースレンダリング) です。
「エネルギー保存則」などの物理法則に基づいた計算を行うことで、どんな照明環境（昼の太陽の下でも、夜の暗い部屋でも）でも、その物体が現実的に見えるように設計されています。

プログラマにとってのマテリアルとは、結局のところ **「フラグメントシェーダーのコード」** そのものです。どの数値を掛け合わせ、どのテクスチャをどう合成するか。そのロジックの積み重ねが、画面上の質感を生み出します。

### 低レイヤーの現実：GPUメモリの圧迫

テクスチャは、グラフィックスメモリ（VRAM）を最も消費する要因です。
高解像度な4Kテクスチャを大量に読み込めば、すぐにメモリはパンクし、アプリはクラッシュします。

これを防ぐために、プロの現場では **「テクスチャ圧縮」** という特殊な技術が使われます。通常のJPGやPNGと違い、GPUが圧縮された状態のまま直接読み込める形式（BC, ASTCなど）を使います。このあたりの最適化は、ハードウェアの仕組みを深く理解していないと手が出せない領域です。

### まとめ

- テクスチャは色だけでなく、凹凸や質感の「データ」を運ぶ。
- UV座標とサンプリングによって、モデルの表面が彩られる。
- PBR（物理ベースレンダリング）が、現実的な質感のスタンダード。
- 質感の美しさと、メモリ消費量のトレードオフを常に意識する。

さて、物体の質感が整ったら、次は「文字」という、グラフィックスプログラミングにおける意外な難所に挑んでみましょう。
