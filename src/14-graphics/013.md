# テキスト描画の極意：SDFとMSDF

3Dグラフィックスの世界に足を踏み入れた初心者が、最初の方にぶつかる大きな壁があります。それは **「画面に文字（テキスト）を表示する」** という、一見すると簡単そうな作業です。

通常のWeb開発なら `<div>` タグを書くだけで済みますが、GPUのキャンバス（WebGLやWebGPU）に直接、きれいで高速な文字を描くのは、実は非常に高度な技術を要する難所なのです。

### なぜ文字は難しいのか？

文字（フォント）は、本質的には「ベクターデータ（輪郭線の集まり）」です。しかし、GPUは三角形を描くのが得意で、複雑な曲線を塗りつぶすのは苦手です。

最も単純な方法は、文字を画像にして貼り付ける「ビットマップフォント」ですが、これには弱点があります。カメラが文字に近づいて拡大されたとき、ピクセルが引き伸ばされてボロボロにボケてしまうのです。かといって、巨大な解像度の画像を用意すれば、ビデオメモリを使い果たしてしまいます。

### SDF：輪郭からの距離で描く

この問題を解決するために、Valve社（ゲーム配信プラットフォームSteamの運営元）が発表した画期的な手法が **SDF** (*Signed Distance Field* / 符号付き距離場) です。

SDFの考え方は非常にユニークです。「文字の形」そのものを保存するのではなく、**「ある地点が、文字の輪郭からどれくらい離れているか」** という距離のデータを画像に保存します。

- 輪郭の内側ならプラスの距離。
- 輪郭の外側ならマイナスの距離。

フラグメントシェーダーの中で、「距離が0以上の場所だけ色を塗る」という処理を一行書くだけで、文字が描画されます。この方法の素晴らしい点は、どれだけ拡大しても、数学的な計算によって常に「シャープな輪郭」が再現されることです。

### MSDF：多色の距離場でさらに高精細に

SDFには一つ弱点がありました。漢字のような複雑な形の文字や、鋭い角（コーナー）がある文字を拡大すると、角が丸まってしまう現象です。

これを改良したのが **MSDF** (*Multi-channel SDF*) です。
赤・緑・青の3つのチャンネルにそれぞれ異なる距離情報を書き込み、それらをシェーダーで組み合わせて判定します。これにより、従来のSDFよりもさらに高精細で、鋭い角を維持したまま文字を描画できるようになりました。現代の高品質な3DゲームやUIエンジンの多くは、このMSDFを標準的に採用しています。

### テキスト描画の「低レイヤー」な設計

文字を表示するには、単に形を描くだけでなく、以下の設計が必要です。

- **フォントアトラス**: 必要な文字を一千個、一万個と大きな一枚の画像に詰め込む。
- **カーニング/レイアウト**: 文字ごとの幅や、文字同士の間隔を正しく計算して、頂点バッファを生成する。

これらの処理は非常に複雑なため、Hydentのような新しい言語やエンジンを開発する際には、いかに効率的で美しいテキストエンジンを構築するかが、そのシステムの「格」を決めると言っても過言ではありません。

### 低レイヤーの現実：GPUへの文字データの動的転送

チャットアプリのように、表示する文字が刻々と変わる場合、そのたびにCPU側で新しい頂点データを作り、GPUへ送り直さなければなりません。

この「動的なバッファ更新」は、前章で学んだ通りコストが高い作業です。文字を一文字ずつ描画するのではなく、文章全体を一つのバッファにまとめて一気に描画（バッチング）するなどの工夫が、60fpsを維持するためには不可欠です。

### まとめ

- 文字の描画はGPUプログラミングにおける大きな課題。
- SDFは「距離データ」を使うことで、拡大してもボケないシャープな文字を実現する。
- MSDFはSDFをさらに進化させ、鋭い角や複雑な形状を美しく再現する。
- パフォーマンスのために、フォントアトラスやバッチングといったシステム設計が必要。

「文字がきれいに出る」という当たり前のことが、実はエンジニアの血の滲むような工夫の上に成り立っていることが分かったでしょうか。さて、静止画としての描画技術は十分です。次は、これらの物体を「時間とともに動かす」ための心臓部、ゲームループについて学びましょう。
