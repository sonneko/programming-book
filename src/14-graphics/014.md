# ゲームループ：リアルタイムシステムの心臓

これまでのプログラミング（Webアプリやバッチ処理など）の多くは、「ユーザーがボタンを押したとき」や「URLにアクセスしたとき」にだけ処理が走る **イベント駆動型** でした。しかし、3Dグラフィックスの世界、特にビデオゲームやシミュレーションでは、プログラムは **「常に動き続ける」** 必要があります。

この、終わることのない無限の繰り返しの構造を **メインループ**、あるいは **ゲームループ** (*Game Loop*) と呼びます。

### ループの三拍子：Input, Update, Draw

基本的なゲームループは、1秒間に60回（またはそれ以上）、以下の3つのステップを超高速で繰り返します。

1. **入力処理 (Process Input)**:
   キーボード、マウス、ゲームパッドの状態を読み取ります。「今、右矢印キーが押されているか？」を確認する瞬間です。
2. **状態更新 (Update)**:
   前のフレームから今までの間に、世界で何が起きたかを計算します。キャラクターを歩かせ、敵のAIを動かし、弾が壁に当たったかを判定します。
3. **描画 (Draw)**:
   更新された世界の最新の状態をもとに、GPUに描画命令を送り、画面を書き換えます。

ブラウザ環境では `requestAnimationFrame` というAPIを使ってこのループを予約します。これは、ディスプレイの更新タイミング（垂直同期）に完璧に合わせて関数を呼び出してくれる、ブラウザが提供する最高のタイマーです。

### デルタタイム：時間の「不平等」を解決する

ゲームループを作る上で、絶対に忘れてはならないのが **デルタタイム** (*Delta Time*) です。

もし、「1フレームごとに1ピクセル進む」というプログラムを書いたとしましょう。
- 性能の良いPC（120fps）では、1秒間に120ピクセル進みます。
- 性能の悪いPC（30fps）では、1秒間に30ピクセルしか進みません。

これではゲームになりません。これを解決するために、前回のループから今回のループまでにかかった秒数（デルタタイム）を計算し、移動距離に掛け算します。

```javascript
// 正しい移動の計算
position.x += velocity.x * deltaTime;
```

これにより、フレームレートがどう変動しようとも、現実の時間軸上ではすべてのプレイヤーが同じ速度で動く「時間の平等」が保たれます。

### 固定時間ステップの重要性

さらに高度なシステム（特に物理演算）では、描画は可変（デルタタイム）で行い、物理計算だけは「必ず0.01秒ごとに1回」といった **固定時間ステップ** (*Fixed Timestep*) で実行する手法がとられます。これは、物理演算が非常に繊細で、計算の間隔がバラバラだとシミュレーションが破綻（爆発）してしまうことがあるからです。

### 低レイヤーの現実：ブロッキングとジッター

もし `Update` や `Draw` の処理が重すぎて、16.6ms（60fpsの限界）を超えてしまったらどうなるでしょうか？ 画面は一瞬止まり、ユーザーは「カクつき」を感じます。これを **ジッター** や **スタッター** と呼びます。

エンジニアは、たとえどれほど複雑な計算をしていても、指定された時間内に必ず処理を終わらせ、ブラウザ（OS）に制御を返さなければなりません。この「時間の極限の制約」の中で最高のパフォーマンスを出す戦いこそが、リアルタイムシステム開発の真髄です。

### まとめ

- リアルタイムシステムは、Input → Update → Draw の無限ループで動く。
- ブラウザでは `requestAnimationFrame` を活用する。
- デルタタイムを使うことで、ハードウェアの性能差を吸収する。
- 物理演算などは、安定性のために固定時間ステップで行う。
