# 抽象クラス・継承

オブジェクト指向プログラミングにおいて、「再利用性」と「秩序」を両立させるための強力な道具が、**継承**（*Inheritance*）と**抽象クラス**（*Abstract Class*）です。

### 抽象クラス：未完成の設計図

抽象クラスとは、一言で言えば「それ自体では実体（インスタンス）を作れない、未完成のクラス」です。`abstract class` というキーワードを使って定義します。

なぜわざわざ未完成のクラスを作るのでしょうか？ それは、複数のクラスに共通する「性質」や「ルール」をひとまとめにし、後から作る具体的なクラス（派生クラス）にそのルールを守らせるためです。

```typescript
// 抽象クラス：動物という概念を定義
abstract class Animal {
  constructor(public name: string) {}

  // 抽象メソッド：鳴き声。具体的な鳴き方は後で決める。
  abstract makeSound(): void;

  // 共通のメソッド：移動。これはどんな動物も共通なので実装まで行う。
  move(): void {
    console.log(`${this.name} は移動しています...`);
  }
}

// Animalを継承して「犬」を作る
class Dog extends Animal {
  // abstract makeSound() を実装することを強制される
  makeSound() {
    console.log('ワンワン！');
  }
}

// const a = new Animal("謎の生物"); // エラー！抽象クラスは実体化できない
const myDog = new Dog("ポチ");
myDog.makeSound(); // ワンワン！
myDog.move();      // ポチ は移動しています...
```

### テンプレートメソッドパターン：枠組みだけ決める

抽象クラスは「テンプレートメソッドパターン」と呼ばれる設計手法によく使われます。これは、「全体の処理の流れ（テンプレート）」を基底クラスで決めておき、その中の「具体的な細かい処理」だけを派生クラスに任せる、というやり方です。

例えば、「ファイルの読み込み・変換・保存」という一連の流れがあるとします。「読み込み」と「保存」は共通でも、「変換」の仕方はPDFかCSVかで異なります。このような場合、変換処理だけを `abstract` メソッドにすることで、全体の秩序を保ちつつ柔軟に実装を変えることができます。

### 抽象クラス vs インターフェース

インターフェース（`interface`）を学んだ際、「これもルールを強制するものじゃないの？」と思ったかもしれません。確かに似ていますが、使い分けのポイントがあります。

| 特徴 | 抽象クラス (`abstract class`) | インターフェース (`interface`) |
| :--- | :--- | :--- |
| **多重継承** | できない（親は1人だけ） | できる（複数のインターフェースに従える） |
| **中身の実装** | 書ける（共通ロジックを共有できる） | 書けない（名前と型だけ決める） |
| **考え方** | 「AはBの一種である」という親子関係 | 「Aは〜という機能を持っている」という契約 |
| **メンバ** | 変数やメソッドの実体を持てる | 型の定義しか持てない |

「犬は動物である（Animal）」という親子関係なら抽象クラス、「このクラスはシリアライズ（データ保存）可能である（Serializable）」という追加機能のようなルールならインターフェースを使うのが一般的です。

### 継承の落とし穴：深い階層は危険

継承は便利ですが、使いすぎると毒になります。親の親、そのまた親...と継承の階層が深くなると、「親クラスをちょっと直したら、全然関係ない孫クラスでバグが出た」という悲劇（密結合）が起こります。

最近のプログラミングでは、**「継承よりも委譲（Composition over Inheritance）」**という考え方が主流です。何でもかんでも親から引き継ぐのではなく、必要な機能を持っている別のクラスを自分のプロパティとして持つことで、より柔軟な設計が可能になります。

継承は「本当に、絶対に親子関係である」と言い切れる場合にのみ使う。これが、美しいコードを書くための秘訣です。
