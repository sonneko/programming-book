# ESmoduleとCommonJS：モジュールシステムの歴史と現代の正解

JavaScriptを学んでいると、コードを別のファイルに分割して読み込む際に、二通りの書き方に出会うはずです。一つは `require` を使う方法、もう一つは `import` を使う方法です。

「なぜ二つもあるの？」と混乱するかもしれませんが、これはJavaScriptが歩んできた苦難の歴史の産物です。TypeScriptで開発を行う上でも、この二つの違いを理解しておくことは、トラブルを回避するために非常に重要です。

### 1. CommonJS (CJS)
Node.jsが誕生した当初、標準として採用されたのが **CommonJS** です。

```javascript
// エクスポート側
const add = (a, b) => a + b;
module.exports = { add };

// インポート側
const { add } = require('./math');
```

CommonJSは、読み込みが「同期的（実行が終わるまで待つ）」に行われるのが特徴です。これはサーバーサイド（Node.js）では問題ありませんでしたが、ネットワークを介してファイルを読み込むブラウザ環境では、読み込み待ちが発生して画面が固まってしまうため、相性が良くありませんでした。

### 2. ES Modules (ESM)
JavaScriptの言語仕様（ECMAScript）として、公式に定義された現代的なモジュールシステムが **ES Modules** です。

```typescript
// エクスポート側
export const add = (a: number, b: number) => a + b;

// インポート側
import { add } from './math';
```

ESMは、読み込みが「静的」かつ「非同期」に行われます。これの何が嬉しいかというと、プログラムを動かす前にどのファイルが何を必要としているかを完璧に把握できるため、ViteやWebpackといった「バンドラー」と呼ばれるツールが、使っていないコードを削ぎ落とす（ツリーシェイキング）といった強力な最適化を行えるようになります。

### TypeScriptでの「正解」

現代のTypeScript開発においては、**ES Modules (ESM)** を使うのが圧倒的な正解です。

TypeScriptコンパイラは、あなたの書いた最新の `import/export` 構文を、設定次第で CommonJS にも ESM にも変換できます。この挙動を制御するのが `tsconfig.json` です。

```json
{
  "compilerOptions": {
    "module": "ESNext",         // 最終的にどの形式で出力するか
    "moduleResolution": "node", // インポート先のファイルを探すアルゴリズム
    "esModuleInterop": true     // ESMとCommonJSの混在を許容するおまじない
  }
}
```

### なぜ今さら CommonJS を知る必要があるのか？

「ESMが正解なら、CommonJSは忘れればいいのでは？」と思うかもしれません。しかし、現実はそう甘くありません。

あなたがnpmでインストールしようとする古いライブラリの多くは、まだ CommonJS で書かれています。これらを ESM のコードから読み込もうとすると、時としてエラー（「Default Exportがありません」など）が発生します。その際に、背後で二つのシステムが衝突しているという知識があれば、冷静に対処できるはずです。

### まとめ：未来へ進むために

1.  自分のコードを書くときは、常に **ES Modules (`import/export`)** を使う。
2.  Node.jsの新しいバージョンでは、`package.json` に `"type": "module"` と書くことで ESM がデフォルトになる。
3.  古いライブラリとの互換性に困ったら、`esModuleInterop` 設定を確認する。

歴史を知ることは、未来のトラブルを解決する力になります。
