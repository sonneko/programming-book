# オブジェクト指向の用語整理：迷子にならないための羅針盤

オブジェクト指向を学び始めると、呪文のような専門用語が次々と登場し、混乱してしまうことがあります。しかし、これらは単なる言葉遊びではなく、エンジニア同士が複雑な設計について議論するための「共通言語」です。

ここで一度、TypeScriptの文脈に沿って、重要な用語を整理しておきましょう。

### 1. 基本的な要素
- **クラス (Class)**: オブジェクトの「設計図」です。どんなデータ（プロパティ）を持ち、どんな振る舞い（メソッド）をするのかを定義します。
- **インスタンス (Instance)**: 設計図から実際に作られた「実体」です。`const myCar = new Car()` としたときの `myCar` がインスタンスです。
- **コンストラクタ (Constructor)**: インスタンスが生成される瞬間に実行される、初期化用の関数です。

### 2. オブジェクトの構成要素
- **プロパティ (Property)**: オブジェクトが持つ「属性」や「状態」です。クラスの中の変数だと考えてください。
- **メソッド (Method)**: オブジェクトができる「動作」や「計算」です。クラスの中の関数です。
- **フィールド (Field)**: プロパティとほぼ同義ですが、特にクラス内で宣言された変数を指してこう呼ぶことが多いです。

### 3. 三大要素（＋α）
- **カプセル化 (Encapsulation)**: 内部のデータを隠し、外部からは許可されたメソッドのみを通じて操作させることです。`private` や `protected` といったアクセス修飾子を使って実現します。
- **継承 (Inheritance)**: あるクラスの機能をベースにして、新しいクラスを作ることです（`extends`）。「親クラス（基底クラス）」の性質を「子クラス（派生クラス）」が引き継ぎます。
- **ポリモーフィズム (Polymorphism / 多態性)**: 同じ名前のメソッドを呼び出しても、相手のクラスによって中身が切り替わることです。呼び出し側は相手の正体を詳しく知らなくても、期待通りの動作をさせることができます。
- **抽象化 (Abstraction)**: 複雑な内部実装を隠し、必要なインターフェース（窓口）だけを見せることです。抽象クラス（`abstract class`）やインターフェース（`interface`）がこの役割を担います。

### 4. 設計の道具
- **インターフェース (Interface)**: 「このメソッドとこのプロパティを必ず持っていなさい」という「契約（ルール）」です。クラスに `implements` させることで、一定の品質を保証します。
- **アクセス修飾子 (Access Modifiers)**: `public`（誰でもOK）、`private`（自分だけ）、`protected`（自分と子供だけ）といった、見える範囲（スコープ）を制限するキーワードです。

### 用語の海を泳ぎ切るために

これらの用語をすべて一度に暗記する必要はありません。大切なのは、コードを書いている最中に「あ、これはポリモーフィズムを使っているな」とか「ここはカプセル化が甘いからバグになりそうだな」と、学んだ概念と現実のコードを紐付けることです。

特にTypeScriptでは、**「インターフェース」**を使いこなせることが脱・初心者への近道です。実装（どうやるか）よりも契約（何をやるか）を重視する姿勢こそが、美しいオブジェクト指向設計の第一歩となります。
