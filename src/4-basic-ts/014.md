# 非同期処理

JavaScript（および TypeScript）は、基本的に **「シングルスレッド」** で動作します。これは、一度に一つのことしかできない、極めて生真面目な実行エンジンであることを意味します。

しかし、現代の Web 開発では、「ネットワーク通信」「ファイルの読み書き」「データベースへのアクセス」といった、完了までに時間のかかる処理が頻繁に登場します。もしこれらを「同期的に（順番に）」実行してしまうと、処理が終わるまで画面がフリーズし、ボタンも押せず、スクロールもできない最悪のユーザー体験を招いてしまいます。

これを解決するための魔法の仕組みが **「非同期処理」** です。

## 1. 非同期処理の進化：地獄から楽園へ

JavaScript の歴史は、非同期処理との戦いの歴史でもあります。

### コールバック地獄 (Callback Hell)
最も古い手法です。「処理が終わったら、この関数を呼んでね」と関数を渡します。
```javascript
getData(url, (data) => {
  parseData(data, (parsed) => {
    saveData(parsed, (result) => {
      console.log("ようやく完了！");
    });
  });
});
```
処理が重なるほど右へ右へとコードがズレていき、どこでエラーが起きたか把握するのも困難になります。これが有名な「コールバック地獄」です。

### Promise の登場
この地獄を救うために現れたのが **Promise** です。非同期処理の状態（「まだ途中」「成功！」「失敗……」）を一つの「オブジェクト」として扱えるようにしました。

```typescript
const myPromise = new Promise<string>((resolve, reject) => {
  // ここで重い処理を行う
  const success = true;
  if (success) resolve("成功！");
  else reject("失敗...");
});

myPromise
  .then(res => console.log(res))
  .catch(err => console.error(err));
```
`.then()` で処理を数珠つなぎにできるため、コードが平坦になり、読みやすさが劇的に向上しました。

## 2. Promise の状態変化

Promise には、必ず 3 つの状態のいずれかが割り振られます。

1. **Pending（保留中）**: 処理が終わるのを待っている状態。
2. **Fulfilled（解決済み）**: `resolve()` が呼ばれ、無事に成功した状態。
3. **Rejected（拒否済み）**: `reject()` が呼ばれ、何らかのエラーが発生した状態。

一度状態が決まると（Fulfilled か Rejected になると）、それ以降二度と状態は変わりません。これを **「状態の確定（Settled）」** と呼びます。この「一度決まったら変わらない」という性質（不変性）が、プログラムの予測可能性を支えています。

## 低レイヤーの視点：イベントループとタスクキュー

シングルスレッドなのに、なぜ重い処理をしながら他のことができるのでしょうか？ その秘密は **「イベントループ」** という仕組みにあります。

1. JavaScript が重い処理（`fetch` など）に出会うと、それを OS やブラウザの別のスレッドに放り投げます。
2. JavaScript 本体は、そのまま次の行の実行を続けます。
3. 放り投げた処理が終わると、その結果が **「タスクキュー」** という待ち行列に並びます。
4. メインの処理がすべて終わって暇になった瞬間に、JavaScript はキューから結果を取り出して処理を再開します。

「自分一人で全部やるのではなく、重い荷物は外に任せて、自分は受付窓口に専念する」。この賢い役割分担こそが、JavaScript が爆速で動く理由なのです。

## 私の視点：非同期は「約束」である

Promise という言葉は、直訳すると「約束」です。
「今はまだ結果を持っていないけれど、将来必ず結果（またはエラー）を渡すことを約束するよ」という、未来に対するチケットのようなものです。

この「未来の値を今から扱える」という抽象化は、プログラミングにおいて非常に強力です。
非同期処理をマスターすることは、時間の流れをコントロールする力を手に入れることと同義です。最初は難しく感じるかもしれませんが、イベントループの図解を一度頭に叩き込んでしまえば、これほど心強い味方はありません。

### まとめ

- JavaScript はシングルスレッドなので、画面をフリーズさせないために非同期処理が必須。
- コールバック地獄 ＞ Promise ＞ Async/Await と進化してきた。
- Promise は「Pending」「Fulfilled」「Rejected」の 3 つの状態を持つ。
- イベントループによって、重い処理を「裏側」で実行している。
- 非同期処理は、未来の結果を今から予約する「約束（Promise）」である。

### 展望

Promise の仕組みがわかったら、次はそれをさらに魔法のように使いこなす **`async/await`** 構文について学びましょう。非同期処理を、まるで同期処理（普通のコード）のように書くことができる革命的な機能です。
