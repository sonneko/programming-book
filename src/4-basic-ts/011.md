# 他の言語のオブジェクト指向：TypeScriptとの比較

TypeScriptのオブジェクト指向プログラミング（OOP）を学んでいると、JavaやC#といった「エンタープライズ系」と呼ばれる言語の概念と非常に似ていることに気づくでしょう。これは偶然ではなく、TypeScriptの開発リーダーがC#の設計者でもあるアンダース・ヘルスバーグ氏であることも大きく関係しています。

しかし、TypeScriptはJavaScriptという「動的な海」の上に建つ「静的な城」のような存在です。そのため、伝統的なクラスベース言語とは異なる、独特の面白さがあります。

### JavaやC#との決定的な違い

JavaやC#のような伝統的な言語では、クラスは「型」そのものであり、実行時（ランタイム）にもその型情報が厳格に残ります。これを**「公称型（Nominal Typing）」**と呼びます。例えば、「User」クラスと「Admin」クラスが全く同じプロパティを持っていても、それらは別の型として扱われ、混ぜることはできません。

一方、TypeScriptは**「構造的部分型（Structural Typing）」**という考え方を採用しています。これは、「見た目が同じなら、同じ型として扱って良い」という非常に柔軟なルールです。
「アヒル（インターフェース）のように鳴き、アヒルのように歩くなら、それはアヒルだ」という、いわゆる **ダック・タイピング** を型システムで表現したものです。

### 言語ごとのOOPの特徴

各言語がオブジェクト指向をどのように捉えているか、簡単に見てみましょう。

- **Java / C#**: 純粋なクラスベースOOP。すべてがクラスから始まります。「カプセル化」や「継承」のルールが非常に厳格で、大規模開発において軍隊のような秩序をもたらします。
- **C++**: オブジェクト指向に加え、メモリ管理やテンプレート（ジェネリクス）といった低レイヤーの操作が可能です。「多重継承」という、複数の親クラスを持てる強力かつ危険な機能も持っています。
- **Python / Ruby**: クラスはありますが、非常に動的です。実行中にクラスの定義を書き換えたり、メソッドを後から追加したりすることが可能です（モンキーパッチ）。自由度は高いですが、秩序を保つのは開発者の規律に委ねられます。
- **Go / Rust**: 実は「クラス」という単語が存在しません（Rustには構造体 `struct` があります）。しかし、インターフェースやトレイトという仕組みを使って、オブジェクト指向的な「振る舞いの抽象化」を実現しています。最近のトレンドは、継承を排除し、これらの仕組みを活用する方向にあります。

### TypeScript特有の「シンタックスシュガー」

TypeScriptには、他言語のエンジニアが羨むような便利な書き方があります。

```typescript
class User {
  // コンストラクタ引数に public などを付けるだけで、プロパティの定義と代入が完了！
  constructor(public name: string, private age: number) {}
}

// 伝統的な書き方ならこれだけ必要でした：
// class User {
//   public name: string;
//   private age: number;
//   constructor(name: string, age: number) {
//     this.name = name;
//     this.age = age;
//   }
// }
```

### まとめ：TypeScriptの立ち位置

TypeScriptは、Javaのような「厳格な秩序」と、JavaScriptのような「素早い開発」の、ちょうど良い中間地点に位置しています。

他の言語を学ぶ際、TypeScriptで学んだ「インターフェース」や「継承」の知識はそのまま通用します。しかし、それぞれの言語がどのような「思想」でそれらの道具を採用しているかを知ることで、あなたのエンジニアとしての視座はより高くなるでしょう。
