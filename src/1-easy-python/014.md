# 関数定義

[関数の使用](./005.md)の章でも触れましたが、ここでは「なぜ関数を自分で作るのか」という、より深い設計の視点から関数について考えてみましょう。

## 抽象化という魔法
プログラミングにおいて最も重要な概念の一つに**抽象化（Abstraction）**があります。抽象化とは、複雑な処理の中身を「名前」の後ろに隠して、扱いやすくすることです。

例えば、`print()` 関数が画面に文字を出すために、OSのシステムコールを呼び出し、ビデオメモリを操作している...といった詳細を、私たちは知る必要がありません。ただ「表示してね」と命じるだけでいいのです。

自分で関数を定義することも、これと同じ「魔法」を自ら作り出す行為です。

## DRY原則：繰り返さない
プログラマが最も嫌うことの一つは、同じコードを何度も書くことです。これを避けるための原則を **DRY（Don't Repeat Yourself）** と呼びます。

```python
# 関数を使わない場合
r1 = 10
area1 = 3.14 * r1 * r1
print(f"半径10の面積は {area1}")

r2 = 20
area2 = 3.14 * r2 * r2
print(f"半径20の面積は {area2}")
```

もし、円周率を 3.14159... にもっと詳しくしたいと思ったら、全ての場所を書き換える必要があります。これはミスの元です。

```python
# 関数を定義する場合
def calculate_circle_area(radius):
    return 3.14 * radius * radius

print(f"半径10の面積は {calculate_circle_area(10)}")
print(f"半径20の面積は {calculate_circle_area(20)}")
```

関数にまとめておけば、修正は一箇所で済みます。

## データフローとしての関数
シニアエンジニアの視点に立つと、関数は「一連の動作」ではなく、**「データの変換処理」**に見えます。

1. **入力 (Argument)**: 処理に必要なデータを受け取る。
2. **変換 (Transformation)**: 決まったルールに従ってデータを加工する。
3. **出力 (Return Value)**: 加工した結果を次に渡す。

```python
def tax_included_price(price):
    return int(price * 1.1)
```

この関数は「税抜き価格」という入力を、「税込み価格」という出力に変換している「装置」です。
プログラムはこの「装置」をいくつも連結させて、目的のデータ（出力）を得るための巨大なパイプラインのようなものなのです。

## 良い関数の条件
良い関数とは、**「副作用」が少なく、何をするかが名前だけでわかる**関数です。
- 名前を見ただけで、何を入力して何を返すかが想像できるか？
- 1つの関数が、1つのことだけを正しく行っているか？

これらを意識して関数を定義できるようになると、あなたのコードは劇的に読みやすく、壊れにくくなります。

> ### コラム：関数の「外」と「中」
> 関数の中で作った変数は、その関数の外からは見ることができません。これを変数の**スコープ（Scope）**と呼びます。
>
> ```python
> def my_function():
>     local_val = "秘密のメッセージ"
>     print(local_val)
>
> my_function()
> # print(local_val)  # ここで実行するとエラーになる！
> ```
>
> なぜこうなっているのでしょうか？ それは、関数が呼び出されるたびに、コンピュータがその関数のためだけの専用のメモリ領域（**スタックフレーム**と呼ばれます）を一時的に確保するからです。
>
> 関数が終了すると、その領域はきれいに片付けられ、中の変数も消えてしまいます。この仕組みがあるおかげで、私たちは「他の場所で同じ変数名が使われていないか？」をいちいち心配することなく、安心して関数を自作できるのです。
