# switch文：制御の分岐と最適化

「もし A なら……」「そうでなくて B なら……」「それとも C なら……」。
条件分岐が増えてくると、`if-else if` を延々と書き連ねることになります。コードが縦に長くなり、読みづらくなっていく……そんな時に登場するのが **`switch` 文** です。

`switch` 文は、ある一つの変数の「値」に注目して、処理を枝分かれさせるのに最適な構文です。

## 1. 基本的な構文と break の重要性

まずは、標準的な `switch` 文の形を見てみましょう。

```javascript
const color = 'red';

switch (color) {
  case 'red':
    console.log('止まれ！');
    break;
  case 'yellow':
    console.log('注意！');
    break;
  case 'green':
    console.log('進め！');
    break;
  default:
    console.log('信号の故障です');
}
```

ここで最も重要なのが **`break`** です。これを忘れると、**フォールスルー (Fall-through)** という現象が起きます。
もし `red` のケースで `break` を書き忘れると、コンソールには「止まれ！」だけでなく、次の `yellow` の処理である「注意！」まで表示されてしまいます。JavaScript は `break` に出会うまで、次の `case` を無視して実行し続けるからです。

## 2. 複数のケースをまとめる

フォールスルーを逆手に取って、複数の条件で同じ処理をさせたい場合は以下のように書けます。

```javascript
const day = '土曜日';

switch (day) {
  case '土曜日':
  case '日曜日':
    console.log('お休みです！');
    break;
  case '月曜日':
  case '火曜日':
  case '水曜日':
  case '木曜日':
  case '金曜日':
    console.log('仕事の日です...');
    break;
  default:
    console.log('存在しない曜日です');
}
```

## 3. switch 文の限界：厳密等価判定

`switch` 文の判定は、常に **`===`（厳密等価演算子）** で行われます。つまり、型まで一致していなければなりません。

```javascript
const value = "1";

switch (value) {
  case 1: // 数値の 1
    console.log("数値です");
    break;
  default:
    console.log("不一致です"); // こちらが実行される
}
```

## 低レイヤーの視点：ジャンプテーブルによる高速化

なぜ `if-else if` ではなく `switch` を使うのでしょうか？ 単に見やすいからだけではありません。実は、多くのプログラム実行エンジンにとって、`switch` は最適化しやすい構造なのです。

ケースの数が多い（たとえば 10 個以上など）場合、実行エンジンは「ジャンプテーブル」というものを作成します。
`if` 文の場合、上から順番に一つずつ「これは真か？」とチェックしていく必要がありますが、`switch` なら値をもとに計算を行い、一気に該当する処理のアドレスへ「ジャンプ」できます。

つまり、選択肢が非常に多い場合、`switch` 文の方が実行速度が速くなる可能性があるのです。

## 私の視点：オブジェクトリテラルによる代替

現代の JavaScript 開発では、複雑な `switch` 文を「データ」として扱う手法がよく好まれます。

```javascript
// オブジェクトを使ったマッピング
const signalMessages = {
  red: '止まれ！',
  yellow: '注意！',
  green: '進め！',
};

const color = 'red';
// switch を使わずにメッセージを取り出す
console.log(signalMessages[color] || '信号の故障です');
```

この書き方（ルックアップテーブル）のメリットは、処理（ロジック）とデータ（メッセージ）を分離できることです。条件が増えてもオブジェクトに 1 行足すだけですし、コード全体のネストも浅くなります。

「この `switch` 文、長すぎないかな？」と思ったら、それはオブジェクトや Map を使うべきサインかもしれません。

### まとめ

- `switch` 文は、特定の「値」による多方向の分岐に適している。
- `break` を忘れると、下のケースまで実行される「フォールスルー」が起きる。
- 判定は型まで含めた `===` で行われる。
- 内部的には「ジャンプテーブル」によって高速化されることがある。
- 複雑な分岐は、オブジェクトリテラルに置き換えることも検討しよう。

### 展望

制御構造をマスターしたら、次はそれらの処理で扱う「データの集まり」――コレクション型（配列とオブジェクト）の世界を深く学んでいきましょう。
