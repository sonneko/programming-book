# 基本の制御構造：プログラムの流れを操る

プログラムがただ上から下へと流れるだけなら、それは単なる「指示書」に過ぎません。JavaScript（そしてあらゆるプログラミング言語）の真骨頂は、状況に応じて処理を分けたり、同じことを何度も繰り返したりする **「制御構造」** にあります。

Python を学んだ皆さんなら、`if` や `for` の概念はすでに知っているはずです。ここでは JavaScript 特有の書き方や、エンジニアとして意識すべき「思考の型」について深掘りしていきましょう。

## 1. 条件分岐：if 文と真偽値の「闇」

JavaScript の `if` 文は、基本的な見た目は Python と似ていますが、カッコ `()` と波カッコ `{}` を使うのが特徴です。

```javascript
const score = 85;

if (score >= 90) {
  console.log('秀');
} else if (score >= 80) {
  console.log('優');
} else {
  console.log('可');
}
```

ここで注意したいのが、JavaScript 特有の **「真偽値の判定ルール（Truthy / Falsy）」** です。JavaScript では、`true/false` 以外の値も、`if` 文の中では自動的にどちらかに分類されます。

- **Falsy (偽とみなされるもの)**: `false`, `0`, `""`（空文字）, `null`, `undefined`, `NaN`
- **Truthy (真とみなされるもの)**: 上記以外すべて（`[]` や `{}` も真です！）

## 2. 三項演算子：1 行でスマートに書く

簡単な条件分岐なら、`if` 文を使わずに **「三項演算子」** を使うのが JavaScript 流です。

```javascript
const age = 20;
const status = age >= 18 ? "成人" : "未成年";
```

非常に便利ですが、複雑な条件を 1 行に詰め込みすぎると、一気に読みづらくなります。

```javascript
// 読みにくい例
const result = a > b ? (c > d ? "A" : "B") : "C";

// 読みやすくする工夫
const result = (a > b) 
  ? (c > d ? "A" : "B") 
  : "C";
```

「短く書くこと」よりも「後から読んだときに 1 秒で理解できること」を優先しましょう。

## 3. ループ：for 文と「関数型」の波

JavaScript にも Python のような `for` 文がありますが、現代のフロントエンド開発では、手続き的な `for` 文よりも **「配列のメソッド」** を使うことが圧倒的に多いです。

```javascript
const fruits = ["apple", "banana", "orange"];

// 古い書き方（手続き型）
for (let i = 0; i < fruits.length; i++) {
  console.log(fruits[i]);
}

// 現代的な書き方（宣言型/関数型）
fruits.forEach(fruit => {
  console.log(fruit);
});
```

なぜメソッドを使うのか？ それは、「今何番目か（変数 `i`）」という管理をブラウザ側に任せられるからです。人間が管理する変数が減れば減るほど、バグが入り込む隙間も減っていきます。

## 低レイヤーの視点：分岐予測とパイプライン

コンピュータの CPU レベルで見ると、`if` 文（分岐命令）は非常にコストの高い処理です。

現代の CPU は、命令を先読みして実行する「パイプライン」という仕組みを持っています。しかし、`if` 文があると「次はどっちの命令を読み込めばいいか」が確定しません。
そのため CPU は「たぶんこっちだろう」と予測して実行を進めますが、予測が外れると、先読みした計算をすべて捨ててやり直すことになります（これを **パイプラインストール** と呼びます）。

効率的なコードを書くということは、単に文字数を減らすことではなく、CPU にとって「予測しやすい」論理構造を作るということでもあるのです。

## 私の視点：ガード節で「ネスト」を破壊する

良いコードを書くコツとして、私はいつも **「ガード節（Guard Clause）」** を推奨しています。

```javascript
// 悪い例：どんどん右にズレていく（ネストが深い）
function checkUser(user) {
  if (user) {
    if (user.isActive) {
      if (user.isAdmin) {
        doSomething();
      }
    }
  }
}

// 良い例：ダメなパターンを先に追い出す
function checkUser(user) {
  if (!user) return;
  if (!user.isActive) return;
  if (!user.isAdmin) return;

  doSomething();
}
```

ネスト（波カッコの深さ）が深いコードは、人間の脳のメモリを大量に消費します。条件を満たさないケースをさっさと `return` して追い出すことで、メインの処理が常に「左端」に並ぶようになり、読みやすさが劇的に向上します。

### まとめ

- JavaScript の `if` 文はカッコ `{}` を使う。
- Truthy / Falsy のルール（特に `[]` や `0`）に注意。
- 三項演算子は、可読性が落ちない範囲で使う。
- ループは `forEach` や `map` などのメソッドを使うのが現代流。
- CPU の「分岐予測」を意識した、予測しやすいロジックを心がける。
- ガード節を使ってネストを浅く保つ。

### 展望

条件分岐がわかったら、次はたくさんの分岐をきれいに整理するための `switch` 文、そしてそれを超えた「オブジェクトによるマッピング」というテクニックについて学んでいきましょう。
