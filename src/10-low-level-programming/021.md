# Cの配列

C言語における配列を理解することは、コンピュータのメモリ操作の「裸の姿」を見ることと同義です。

### 配列とポインタの境界線
C言語において、配列名はその配列の「先頭要素のアドレス（ポインタ）」として振る舞います。
```c
int arr[5] = {10, 20, 30, 40, 50};
int *p = arr; // arrは &arr[0] と同じ意味
```
`arr[i]` と書くのと、`*(arr + i)` と書くのは、コンパイラにとっては全く同じ意味です。これは、配列へのアクセスが「ベースアドレスからのオフセット計算」であるというハードウェアの仕組みをそのまま言語に反映しているからです。

### 境界チェックの不在
C言語は、配列の範囲外にアクセスしてもあなたを止めません。
`arr[10]`（存在しない11番目の要素）にアクセスしようとすると、C言語は黙ってその場所にあるメモリを読み書きします。これが「バッファオーバーラン」の正体であり、多くのセキュリティホールや謎のクラッシュの原因です。
高レベル言語はこのチェックを自動で行ってくれますが、C言語は「プログラマは自分が何をしているか分かっているはずだ」という冷徹な信頼（あるいは放任）の上に成り立っています。

### 多次元配列の現実
`int matrix[2][3]` のような多次元配列も、メモリ上では単なる「一本の平らな並び」です。
コンピュータには「次元」という概念はありません。単に `row * width + col` という計算式で、多次元を一本の線に「射影」しているだけです。

### 設計思想：固定長と静的確保
関数の外で宣言された配列（静的配列）は、プログラムのバイナリの中に最初から領域が確保されます。これは実行時の確保コストがゼロであることを意味します。一方で、関数の内側で大きな配列を宣言すると、前述の「スタック領域」を食いつぶす危険があります。

### データの変形
Cの配列は、単なるビットの羅列です。
「この100バイトの配列を、最初は画像データとして扱い、次は通信用のパケットとして解釈する」といった、型に縛られない自由なデータの変形が可能ですが、それは常に破壊の危険と隣り合わせです。
