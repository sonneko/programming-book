# コンパイラによる最適化

コンパイラは、あなたの書いたコードをそのまま機械語にするわけではありません。意味（入力と出力の関係）を変えずに、より効率的に動くようにコードを「作り変え」ます。

### 代表的な最適化の手法
1. **インライン展開**: 小さな関数の呼び出しを、その中身に置き換える。関数の呼び出しコスト（レジスタの待避など）がゼロになります。
2. **ループ展開 (Loop Unrolling)**: ループの回数を減らし、中身の処理を並べる。CPUの「分岐予測」が効きやすくなります。
3. **デッドコード削除**: 決して実行されないコードや、計算しても使われない値を消し去る。
4. **定数畳み込み**: `2 + 3` のような計算を、コンパイル時に `5` に置き換えてしまう。

### 最適化の代償
最適化を強くかけると、コンパイル時間が長くなり、バイナリサイズが大きくなることがあります。また、ソースコードと機械語の対応関係がバラバラになるため、デバッグが非常に難しくなります。

### 低レイヤーの現実：コンパイラを「信じすぎない」
現代のコンパイラは超人的ですが、プログラマが「意図」を明確に伝えないと最適化できないことがあります。
例えば、二つのポインタが「同じ場所を指している可能性がある（エイリアシング）」と、コンパイラは安全のために最適化を諦めます。C言語の `restrict` キーワードなどは、この情報をコンパイラに伝えて最適化を促すための道具です。

### 設計思想：まずは「正しさ」、次に「計測」
「最適化は諸悪の根源である」という有名な言葉があります。
最初から複雑な最適化を意識して読みにくいコードを書くのではなく、まずは「正しく、意図が伝わる」コードを書くべきです。その上で、実際にボトルネックとなっている場所を「計測（プロファイリング）」し、そこだけをコンパイラが助かるような書き方に修正するのが、一流の仕事です。

### データの変形フロー
最適化とは、計算の「経路」を短縮する作業です。
同じ目的地（出力）にたどり着くなら、より少ないガソリン（CPUサイクル）と時間で行ける道を探し出す。コンパイラの裏側では、数千ものルールに基づいた「グラフ変形」が行われています。
