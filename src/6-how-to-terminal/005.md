# CLIツールのお作法

ターミナルという「黒い画面」でコマンドを叩くのは、最初は魔法の呪文を唱えているようで怖いかもしれません。しかし、Git をはじめとする多くの **CLI (Command Line Interface)** ツールには、共通の「設計思想」と「お作法」があります。

この流儀を身につけることで、新しいツールに出会ったときも「たぶんこうすれば動くはずだ」という直感が働くようになります。エンジニアとしての「勘」を養うための、CLI の基本ルールを紐解いていきましょう。

## 1. ヘルプとマニュアル：自力で解決する第一歩

すべての優れた CLI ツールは、自分自身の説明書を内蔵しています。

- **`--help` または `-h`**: ほとんどのコマンドで使えます。そのコマンドにどんなオプションがあるかを表示してくれます。
  - 例：`git --help`, `npm install --help`
- **`man` (Manual)**: Linux/Mac では `man <コマンド名>` と打つと、詳細なマニュアルを読むことができます。

「ググる」前にまずヘルプを見る。これが、最新の（まだネットに情報が少ない）ツールを使いこなすプロの習慣です。

## 2. UNIX 哲学：一つのことをうまくやる

CLI ツールの多くは、**「UNIX 哲学」** に基づいて設計されています。

> 「一つのことを行い、またそれをうまくやれ。プログラムが共同して動くように作れ。プログラムがテキスト・ストリームを扱うように作れ。なぜなら、それが普遍的なインターフェースだからだ。」

この哲学を象徴するのが **パイプ (`|`)** です。
あるコマンドの「出力」を、別のコマンドの「入力」として繋ぐことができます。

```bash
# ファイルの一覧 (ls) から、"txt" という文字を含むものだけを探す (grep)
ls | grep ".txt"
```

個々のツールはシンプルでも、それらをパイプで繋ぐことで、無限に複雑なデータ処理パイプラインを即座に構築できるのです。

## 低レイヤーの視点：標準ストリームとファイル記述子

コンピュータの内部（OS カーネル）から見ると、プログラムとのやり取りは **「標準ストリーム (Standard Streams)」** という、文字データの通り道を通じて行われます。

1. **標準入力 (stdin / FD 0)**: キーボードなどからの入力を受け取る。
2. **標準出力 (stdout / FD 1)**: 正常な実行結果を出力する。
3. **標準エラー出力 (stderr / FD 2)**: エラーメッセージを出力する。

ここで面白いのが、**「すべてはファイルである」** という UNIX の考え方です。
画面への表示も、ファイルへの書き込みも、キーボードからの入力も、OS にとってはすべて「バイトが流れるパイプ（ファイル）」として抽象化されています。

だからこそ、`ls > list.txt` と書くだけで、画面に出るはずだった内容をファイルに「リダイレクト」できるのです。

## 私の視点：エイリアスで自分専用の武器を作る

よく使う長いコマンドを何度も打つのは時間の無駄です。そんな時は **「エイリアス (alias)」** を作りましょう。

```bash
# git status を gs だけで呼び出せるようにする
alias gs='git status'
```

これをシェル（bash や zsh）の設定ファイルに書いておけば、あなたのターミナルは、あなたに最適化された最強の武器になります。
「面倒くさい」と感じた瞬間に、それを自動化したり短縮したりする方法を考える。その「面倒くさがり」な性格こそが、プログラマとしての最大の才能ですよ。

### まとめ

- 新しいコマンドに出会ったら、まずは `--help` でオプションを確認。
- `|` (パイプ) を使って、小さなツールを組み合わせて大きな目的を達成する。
- `stdout` (標準出力) と `stderr` (標準エラー) の違いを意識する。
- UNIX 哲学（一つのことをうまくやる）は、現代のソフトウェア設計にも通じる。
- エイリアスを活用して、自分だけの快適な開発環境を構築しよう。
