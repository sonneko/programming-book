# Gitの基本の使い方

Gitの基本的なワークフローを覚えましょう。

### 基本コマンド

1. `git init`: リポジトリの初期化。
2. `git add <file>`: 変更をステージングエリア（コミット準備）に追加。
3. `git commit -m "message"`: 変更をリポジトリに記録。
4. `git push`: リモートリポジトリ（GitHub等）に反映。

```bash
git add .
git commit -m "機能追加"
git push origin main
```

### Gitの状態遷移とステージングエリア
Gitには「作業ディレクトリ」「ステージングエリア（Index）」「リポジトリ」の3つの領域があります。
`git add` は作業ディレクトリからステージングエリアへ変更をコピーし、`git commit` はステージングエリアの内容をスナップショットとしてリポジトリに保存します。この「ステージング」のステップがあることで、コミット内容を細かく制御できるのです。
### 詰まりポイントと解決策

- **不要なファイルの追加**: `git add .` で一時ファイルや不要なファイルまでステージングしてしまう。`git status` で追加対象を確認してから `git add` する癖を。
- **コミットメッセージ**: 何をしたか不明なコミットは後で困る。`git commit -m` の際は「何をしたか」を端的に記述する。
- **Gitの「取り消し」**: 失敗した場合は焦らず `git status` を見る。多くの場合、解決策が提案されています。

### 技術的な深み：HEADとブランチ

Gitにおける「ブランチ」とは、単なる特定のコミットを指し示す「ポインタ（参照）」に過ぎません。`HEAD` は現在作業中のコミットまたはブランチを指す特殊なポインタです。

```text
[コミットA] <- [コミットB] <- [コミットC (HEAD/main)]
```

この状態で新しいコミットを行うと、`main` ポインタが新しいコミットを指すように進みます。Gitの強力さは、これらのポインタを自由に動かすことで実現されるコミットグラフの柔軟性にあります。

この「ステージング」→「コミット」→「プッシュ」の流れがGit開発の基本ですが、各ステップでデータがどこからどこへ移動し、どのように変形されているかを常に意識してください。
