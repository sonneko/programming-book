# 再利用性・設計のキソ

プログラミングができるようになることと、良いプログラムを書けるようになることの間には、大きな壁があります。その壁を乗り越えるための鍵が **「再利用性（*Reusability*）」** と **「設計（*Design*）」** という考え方です。

今回は、これまでに学んだ「変数」「リスト」「繰り返し」を使って、どのように「筋の良いコード」を書いていくべきかを考えてみましょう。

## 比較：硬いコードと柔らかいコード

まずは、あるリストの合計金額を計算するプログラムを見てください。

```python
# パターンA
prices = [100, 200, 300]
total = 0

for price in prices:
    total = total + price

print(f"合計金額は {total} 円です")
```

このプログラムの凄いところは、`prices` リストの中身が 3 個だろうが 100 個だろうが、中身が 100 円だろうが 1 億円だろうが、**下の計算ロジック（`for`文の部分）を一切書き換えずに動く** という点です。

次に、同じ結果を出す別の書き方を見てみましょう。

```python
# パターンB
total = 100 + 200 + 300
print(f"合計金額は {total} 円です")
```

結果は同じですが、もし「商品が 4 つに増えた」としたらどうでしょう？ パターン B では `total = 100 + 200 + 300 + 400` と、計算式そのものを書き換える必要があります。

これを、私は「硬いコード（パターンB）」と「柔らかいコード（パターンA）」と呼んでいます。

## ロジックとデータの分離

パターン A が「柔らかい」のは、**「計算のやり方（ロジック）」** と **「計算される対象（データ）」** が明確に分離されているからです。

- **データ**: `prices = [100, 200, 300]`
- **ロジック**: `for`文による合計計算

この分離ができていると、プログラムの一部を変更（データの更新）しても、他の部分（ロジック）に影響を与えません。これが大規模なソフトウェア開発における「保守性（メンテナンスのしやすさ）」の基礎になります。

## 抽象化：具体的なことを忘れる

プログラミングにおける「抽象化（*Abstraction*）」とは、**「具体的なディテール（100円、200円...）を一度忘れて、共通のルール（リストの中身を全部足す）だけを取り出す」** 作業のことです。

例えば、「リンゴの価格リストを足すプログラム」と「電車の運賃リストを足すプログラム」を別々に作るのは無駄ですよね。どちらも「数値のリストを足す」という点では同じです。

この「共通のルール」に名前をつけて、どこでも使い回せるようにしたのが、次に詳しく学ぶ **「関数」** です。

## 低レイヤーの視点：コードの局所性（Locality）

「なぜわざわざ変数を分けたり、抽象化したりするの？ 直接計算した方が速いんじゃない？」と思うかもしれません。確かに、極限のパフォーマンスが求められる場面では、直接的な記述が有利なこともあります。

しかし、現代の CPU やコンパイラにとって、整理されたコードはむしろ最適化しやすい対象です。

整理されたコードは、メモリ上の配置が予測しやすくなり（参照の一貫性）、CPU のキャッシュ（*Cache*）を効率的に使えるようになります。逆に、あちこちに具体的な数値が散らばったスパゲッティコードは、CPU にとっても「次に何をすべきか」が予測しづらい、効率の悪いコードになってしまうのです。

## 私の視点：未来の自分への思いやり

私がコードを書くときにいつも自分に問いかけるのは、**「3ヶ月後の自分がこのコードを読んだとき、すぐに理解して修正できるか？」** ということです。

プログラムは、一度書いて終わりではありません。必ず修正や追加の要望が来ます。その時に、具体的な数値が直接書き込まれた「硬いコード」だと、どこを直せばいいか分からず、一つ直すと別の場所が壊れる……という恐怖（デグレード）に怯えることになります。

「再利用性」や「設計」を意識することは、未来の自分を助けるための「思いやり」なのです。

### まとめ

- 「データ」と「ロジック」を分けることで、変化に強いコードになる。
- 具体的な値を直接扱わず、変数やリストを使って「抽象化」する。
- 整理されたコードは、人間にとってもコンピュータにとっても効率が良い。
- 設計の基本は、変化する部分（データ）と変化しない部分（ルール）を見極めること。
