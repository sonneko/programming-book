# Cのコレクション

C言語には、他の言語にあるような便利な「コレクションライブラリ（リスト、マップ、セット等）」は標準で存在しません。

### 「なければ作る」の精神
C言語でリスト（可変長配列）を使いたい場合、あなたは自分で構造体を作り、メモリを確保し、要素を追加する関数を書く必要があります。
```c
typedef struct {
    int *data;
    size_t size;
    size_t capacity;
} Vector;
```
これは面倒に思えるかもしれませんが、教育的な観点からはこれ以上ない教材です。ベクターが内部でどのように `realloc` を呼び、ポインタを更新しているのかを一行ずつ書くことで、データ構造の「コスト」を肌で感じることができます。

### 連結リスト（Linked List）
「メモリ上に連続して並ぶ」配列とは対照的に、「次のデータの場所（ポインタ）をデータ自身が持っている」のが連結リストです。
- **メリット**: 途中の要素の挿入・削除が O(1) でできる。
- **デメリット**: 特定の番目へのアクセスが O(n) になる。また、データがメモリ上に散らばるため、CPUキャッシュが効きにくく、実測速度は配列に完敗することが多いです。

### 低レイヤーの現実：抽象化のコスト
高レベル言語のコレクションは非常に便利ですが、裏側では「あらゆる用途に耐える汎用性」のために、わずかな余分なメモリや計算時間を消費しています。
極限のパフォーマンスが必要なC言語の世界では、その問題に「特化した専用のデータ構造」を自前で構築することで、汎用ライブラリでは到達できない領域を目指します。

### ミクロな要件定義：データ構造の選定
- 検索が多いなら、ハッシュマップやソート済み配列。
- データの追加・削除が激しいなら、連結リストや木構造。
- メモリ効率とキャッシュを最優先するなら、単純な配列。
どのような「データの変形」を行いたいのかによって、最適な「器（コレクション）」を選ぶ。これがエンジニアリングの真髄です。
