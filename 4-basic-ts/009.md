# 抽象クラス・継承

抽象クラス（`abstract class`）は、それ自体をインスタンス化（`new`）することはできません。派生クラス（継承クラス）に対して、特定の構造や実装の強制（コントラクト）を行うための基盤として設計されます。

### 抽象クラスの役割

共通の属性や振る舞いを持つクラス群において、抽象クラスを使用することで、共通のロジックを再利用しつつ、派生クラスに固有の実装を強制できます。

```typescript
abstract class Animal {
  constructor(public name: string) {}

  // 派生クラスで実装を強制する抽象メソッド
  abstract makeSound(): void;

  // 共通のメソッド実装
  move(): void {
    console.log(`${this.name} は移動しています...`);
  }
}

class Dog extends Animal {
  makeSound() { console.log('Woof!'); }
}
```

### テンプレートメソッドパターン

抽象クラスは「テンプレートメソッドパターン」の実装に最適です。基底クラスで大枠のアルゴリズムを定義し、具体的な処理ステップを派生クラスで実装する設計です。これにより、アルゴリズムの全体的な流れを保持しつつ、詳細部分のカスタマイズを派生クラスに委ねることができます。

### 抽象クラスとインターフェースの比較

| 特徴 | 抽象クラス (`abstract class`) | インターフェース (`interface`) |
| :--- | :--- | :--- |
| **多重継承** | 不可 (単一継承のみ) | 可能 (複数実装可) |
| **実装の共有** | 可能 (共通メソッドなど) | 不可 (型定義のみ) |
| **目的** | 「AはBである」という階層関係 | 「Aは〜ができる」という機能能力 |
| **アクセス修飾子** | 使用可能 (`protected` など) | 不可 (すべてpublic) |

### どちらを使うべきか

抽象クラスは、「共通の機能実装」をクラス階層で共有したい場合に適しています。一方、インターフェースは、「クラスがどのような機能を持つべきか（契約）」を定義し、実装の詳細は各クラスに完全に任せたい場合に適しています。

### 継承の注意点とリファクタリング

継承階層が深くなると、クラス間の結合度が高まり、コードの変更が困難になります。インターフェースを優先的に利用し、クラス間の結合度を低く保つ設計が推奨されます。もし抽象クラスが純粋なインターフェースの役割しか果たさなくなったら、インターフェースへ移行するリファクタリングを検討すべきです。
