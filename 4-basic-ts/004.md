# 型注釈：プログラムの宣言的意図

TypeScript を書き始めると、変数名の後ろに `: string` や `: number` といった「おまじない」をたくさん書くことになります。これを **型注釈 (Type Annotation)** と呼びます。

初心者が陥りがちな罠があります。「型注釈は、コンパイラに型を教えるための作業だ」という誤解です。
そうではありません。型注釈は、**「このデータはこうあるべきだ」という、あなた自身の「宣言的意図」をコードに刻み込む行為** なのです。

## 1. 型注釈の基本

JavaScript の変数宣言に、コロン `:` を使って型を付け加えます。

```typescript
// 基本の型
let username: string = 'Alice';
let age: number = 25;
let isPremium: boolean = true;

// 配列の型
let tags: string[] = ['typescript', 'coding'];

// オブジェクトの型 (Interface や Type)
interface User {
  id: number;
  name: string;
}

const user: User = {
  id: 1,
  name: 'Bob'
};
```

これによって、「`age` に文字列を入れようとする」といった、かつて JavaScript で許されていた「曖昧さ」が完全にシャットアウトされます。

## 2. 構造的型付け (Structural Typing)

TypeScript の型システムには、他の言語（Java や C++）とは決定的に違う面白い特徴があります。それが **「構造的型付け」** です。

Java などでは、クラスの名前が一致していないと型が合いません（名前ベース）。
しかし TypeScript では、**「見た目（構造）が同じなら、同じ型とみなす」** というルールを採用しています。

```typescript
interface Point {
  x: number;
  y: number;
}

function printPoint(p: Point) {
  console.log(`x: ${p.x}, y: ${p.y}`);
}

// Point という型を明示していないオブジェクトでも、
// x と y を持っていれば printPoint に渡せる！
const myObj = { x: 10, y: 20, z: 30 };
printPoint(myObj);
```

「カモのように歩き、カモのように鳴くなら、それはカモだ」というこの柔軟な設計思想のおかげで、JavaScript の自由なオブジェクト操作の良さを壊さずに、安全性を手に入れることができているのです。

## 3. 型推論：コンパイラの「忖度」

「すべての変数に型を書くのは面倒だ！」と思うかもしれません。安心してください。TypeScript コンパイラは驚くほど賢いです。

```typescript
// 型を書かなくても、右側の値を見て string だと判断してくれる
let message = "Hello"; 

// ここで数値を入れようとすると怒ってくれる
message = 123; // Error!
```

これを **型推論 (Type Inference)** と呼びます。
「見ればわかること」はコンパイラに任せ、「複雑で間違えやすい部分」や「外部との境界線（関数の引数など）」にはしっかり型注釈を書く。このバランスが、心地よい TypeScript 開発のコツです。

## 低レイヤーの視点：静的解析のアルゴリズム

TypeScript コンパイラが、あなたの書いた型注釈をどうやって検証しているか、少しだけ裏側を覗いてみましょう。

コンパイラは内部で **「データフロー解析」** を行っています。
例えば `if` 文で「もし `user` が `null` でなければ」というチェックを書くと、その `if` ブロックの中では `user` の型から自動的に `null` が消え、安全にアクセスできるようになります。
これを **「型の絞り込み (Type Narrowing)」** と呼びます。

あなたの書いた 1 行のコードの裏で、コンパイラは膨大な数の論理パズルを解き、破綻がないかを確認し続けているのです。

## 私の視点：型は「未来の自分への契約書」

私はよく、型注釈を「契約」に例えます。
「この関数は、文字列をくれたら、数値を返すことを約束するよ」という契約を交わすわけです。

契約があれば、関数の「中」を実装する人は、何が来るか心配しなくて済みます。
関数の「外」で使う人も、何が返ってくるか確信を持って使えます。
一人で開発していても、数ヶ月後の自分は「他人」です。型注釈という契約書を残しておくことで、未来の自分に感謝されること間違いなしですよ。

### まとめ

- 型注釈はコンパイラへの命令ではなく、エンジニアの「意図」の表明。
- 構造的型付けのおかげで、名前ではなく「形」で型を合わせられる。
- 賢い「型推論」を使いこなし、冗長なコードを避けよう。
- 「型の絞り込み」によって、条件分岐に応じた安全性が手に入る。
- 型は開発者同士（あるいは未来の自分）の「契約」である。
