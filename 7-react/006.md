# カスタムフックとロジックの関数合成

React の基本フック（`useState` や `useEffect`）を使えるようになると、コンポーネントの中でいろいろなことができるようになります。しかし、調子に乗っていろいろな処理を書きすぎると、コンポーネントが 100 行、200 行と肥大化し、「何を表示するための部品なのか」が分からなくなってしまいます。

そんな時に真価を発揮するのが **「カスタムフック」** です。
カスタムフックは、React のロジックをコンポーネントから切り出し、**「名前のついた再利用可能な部品」** に変える魔法です。

## 1. カスタムフックとは何か？

一言で言えば、**「React のフックを使っている、普通の JavaScript 関数」** です。
唯一のルールは、名前を **`use`** から始めること。これによって、React の静的解析ツール（リンター）が「あ、これはフックなんだな」と理解し、正しいルールで使われているかをチェックしてくれます。

```tsx
// カスタムフックの例
function useCounter(initialValue = 0) {
  const [count, setCount] = useState(initialValue);
  const increment = () => setCount(prev => prev + 1);
  const decrement = () => setCount(prev => prev - 1);

  // 状態と、それを操作する関数を返す
  return { count, increment, decrement };
}

// コンポーネントでの使用
function App() {
  const { count, increment } = useCounter(10);
  return <button onClick={increment}>{count}</button>;
}
```

コンポーネント側は、`count` がどうやって増えるのかという「詳細（ロジック）」を知る必要がありません。ただフックを呼び出すだけで機能を手に入れられます。

## 2. 責務の分離：見た目と中身を分ける

カスタムフックの最大のメリットは **「責務の分離 (Separation of Concerns)」** です。

- **カスタムフック**: 「データがどう変わるか」「API をどう呼ぶか」という **ロジック** を担当。
- **コンポーネント**: 「データをどう画面に出すか」という **見た目** を担当。

この 2 つを分けることで、コンポーネントは驚くほどスッキリし、宣言的になります。また、同じロジックを別のコンポーネントで使い回すのも簡単です。

## 低レイヤーの視点：クロージャとメモリの効率化

カスタムフックは、内部で `useState` や `useEffect` を呼んでいるだけなので、コンポーネントが再レンダリングされるたびに、フックの関数自体も再実行されます。

ここで重要なのが、カスタムフックの中で定義した関数（上記の `increment` など）も、毎回新しく作り直されているという点です。
ほとんどの場合は問題ありませんが、もしその関数を `useEffect` の依存配列に入れたり、子コンポーネントに渡したりする場合、**「毎回違う関数だと判断される」** ことで、不必要な再レンダリングを引き起こす可能性があります。
そんな時は `useCallback` を使って関数をメモ化（保存）する、といった高度な最適化が必要になります。

## 私の視点：フックは「UI のための API」

私はカスタムフックを、**「コンポーネントのための専用 API」** だと考えています。

たとえば「ウィンドウの横幅を監視するフック」や「ブラウザの LocalStorage を同期するフック」を作っておけば、どんなコンポーネントからでも 1 行でその機能を引き出せます。
「このコンポーネント、コードが長いな……」と感じたら、それは新しいカスタムフックが誕生する合図です。
自分だけの便利なフック集（ツールボックス）を育てていくことは、React エンジニアとしての大きな楽しみの一つですよ。

### まとめ

- カスタムフックは、ロジックを関数として切り出し、再利用可能にする仕組み。
- 名前は必ず `use` から始める。
- 「見た目（コンポーネント）」と「ロジック（フック）」を分けることで保守性が上がる。
- 内部では基本フックを組み合わせて高度な機能（合成）を作れる。
- パフォーマンスが気になる時は `useCallback` や `useMemo` で最適化を。

### 展望

コンポーネントの内部ロジックを整理できるようになったら、次は Web アプリ全体の構造――ページ遷移を司る **「SPA (Single Page Application)」** の世界へと視点を広げましょう。
