# 演習：React 実践アプリケーション開発

7 章で学んだ React と Next.js の知識を総動員して、一つの小さな、しかし本質的なアプリケーションを構築しましょう。
今回のテーマは **「タスク管理（ToDo）アプリ」** です。単純に思えるかもしれませんが、ここには State 管理、コンポーネント分割、副作用の扱いといった、React のエッセンスがすべて詰まっています。

## 演習ミッション：ToDo アプリを完成させよ

以下の 3 つのステップで開発を進めてください。

### ステップ 1：コンポーネントの構築
- **Input エリア**: タスクを入力し、追加ボタンでリストに追加する。
- **List エリア**: 追加されたタスクを一覧表示する。
- **Item エリア**: 各タスク。完了チェックボックスと、削除ボタンを持つ。

「一つの関数（ファイル）に全部書く」のではなく、適切にファイルを分けて作成してください。

### ステップ 2：State の管理（リフティング・ステート・アップ）
- タスクの配列 (`todos`) をどこで管理すべきか考えてください。
- 子コンポーネントから親の State を更新するために、関数を props として渡す練習をしましょう。

### ステップ 3：useEffect による永続化
- ブラウザをリロードしてもタスクが消えないように、**LocalStorage** にデータを保存・取得する機能を `useEffect` を使って実装してください。

## 低レイヤーの視点：State 更新のバッチ処理

タスクを 1 つ追加したとき、React の内部では何が起きているでしょうか。

もし、一つのボタンクリックの中で 3 つの State を同時に更新したとしても、React 18 以降はそれらを **1 回の再レンダリング** にまとめてくれます（バッチ処理）。
これは、ブラウザのリフロー回数を減らし、パフォーマンスを維持するための極めて重要な最適化です。私たちが意識せずとも、React は常に「効率」を追求してくれています。

## 私の視点：設計に「正解」はない

ToDo アプリ一つとっても、作り方は千差万別です。
- 全部のタスクを一つの State で持つか？ 
- それとも完了済みと未完了を分けるか？
- カスタムフック `useTodos` を作るか？

プログラミングの面白さは、**「目的を達成するためのルートが無限にあること」** です。
まずは自分の直感で書いてみて、後から「もっとスッキリ書けないか？」とリファクタリングしてみてください。その試行錯誤の過程こそが、あなたを本物のエンジニアへと成長させます。

### まとめ

- 小さなコンポーネントを組み合わせて大きなアプリを作る（コンポジション）。
- State は「必要最小限の場所」で管理する。
- `useEffect` を使って、外部（LocalStorage など）と同期する。
- DevTools を使いこなし、データの流れを可視化する。
- バッチ処理など、React の最適化の恩恵を理解する。
- 完璧を求めず、まずは「動くもの」を作り、改善を繰り返す。
