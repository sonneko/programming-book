# DOM更新のステップとレンダリングパイプライン

React を使っていると、まるで魔法のように画面がパッパッと切り替わります。しかし、その裏側では、ブラウザのメインスレッドをフリーズさせないための、緻密に計算された「3 つのステップ」が実行されています。

「React が再レンダリングした」という言葉をよく耳にしますが、実はその中身は **「計算」** と **「反映」** の 2 つに大きく分かれているのです。この違いを知ることは、パフォーマンスの良いアプリを作るための必須知識です。

## ステップ 1：レンダーフェーズ (Render Phase)

データ（State）が変わった瞬間に始まる、最初のステップです。

React は、変更されたコンポーネントの関数を再実行し、**「新しい仮想 DOM ツリー」** をメモリ上に作り上げます。
この段階では、**実際の画面（本物の DOM）には 1 ミリも触れていません。**

「計算」だけのフェーズなので、JavaScript の実行速度に依存します。React はここで「前回と今回のツリーを比較して、どこが変わったか」をリストアップします。

## ステップ 2：コミットフェーズ (Commit Phase)

計算が終わったら、いよいよ **「本物の DOM」** に変更を反映させるステップです。

React は、レンダーフェーズで導き出した「最小限の差分」だけをブラウザに伝えます。
「この `div` の色を赤に変えて」「この `li` を削除して」。
これらは一括で行われるため、ブラウザは何度も再描画する必要がなく、非常に効率的です。

## ステップ 3：ブラウザの描画 (Painting)

React の仕事が終わると、今度はブラウザの出番です。
React が書き換えた DOM の情報を元に、ブラウザは **「レイアウト（位置計算）」** と **「ペイント（色塗り）」** を行い、ようやくユーザーの目に光として届きます。

## 低レイヤーの視点：Fiber アーキテクチャと優先順位

以前の React では、この「レンダーフェーズ」が始まると、すべての計算が終わるまでメインスレッドを独占してしまい、ユーザーのクリックや入力が無視される（カクつく）ことがありました。

これを解決したのが、現在の React の心臓部 **「Fiber (ファイバー)」** です。

Fiber は、巨大な計算を小さなタスクに分解します。
「少し計算したら、ブラウザに『何か急ぎの仕事（ユーザーの入力など）はない？』と聞き、あれば一旦計算を中断して入力を優先し、また続きから再開する」。
この **「中断・再開可能なレンダリング」** という高度なスケジューリングによって、どんなに複雑な UI でもサクサクと動き続けることができるのです。

### まとめ

- **レンダーフェーズ**: メモリ上での計算（仮想 DOM の作成と比較）。
- **コミットフェーズ**: 実際の DOM への反映。
- **ペイント**: ブラウザによる画面の描き出し。
- **Fiber**: 計算を細切れにして、ユーザー操作を最優先する仕組み。
- 実際の DOM 操作（コミット）は重いが、計算（レンダー）は非常に速い。
